# تحقيق — Titan Playground

**الحالة:** مكتملة — القرار في [docs/decisions/011-playground.md](../../decisions/011-playground.md)
**تاريخ:** 2026-07-10
**الميزة:** Playground (#5) — من قائمة Planned

---

## 1. المشكلة

**من منظور المطور:**
"أريد أن أفهم كيف يتصرف Titan — كيف يمر update عبر middleware ثم إلى handler، كيف يُبنى Context، كيف تُبنى لوحة مفاتيح — دون أن أبني بوتاً حقيقياً، أو أفتح Telegram، أو أكتب توكن."

المطور الجديد على Titan يتعلم حالياً بطريقتين فقط:
1. قراءة الكود المصدري لـ Core مباشرة.
2. كتابة بوت حقيقي وربطه بـ Telegram الفعلي لتجربة السلوك.

كلاهما مكلف: الأول يتطلب فهم الكود قبل استخدامه، والثاني يتطلب توكن وبيئة تشغيل واتصال شبكة لمجرد فهم كيف يمر update واحد عبر النظام.

**رؤية المستخدم لهذه الميزة (كما وردت):**

> Titan Playground ليس IDE ولا محرر كود. هو بيئة تشغيل وتجريب وفهم لـ Titan.
> الهدف الأساسي: السماح للمطور بتجربة وفهم سلوك Titan بدون بناء مشروع كامل أو كتابة بوت حقيقي.

## 2. ما يوجد حالياً في الـ Core (ذو صلة مباشرة)

| المكوّن | ما يوفره اليوم | الصلة بـ Playground |
|---|---|---|
| `Titan._handle_update(raw_update: dict)` | نقطة الدخول الحقيقية الوحيدة لمعالجة update — تبني `Update`، `Context`، تُطبّق `is_banned`، ثم تمرر عبر `middleware_chain.run()` إلى `dispatch()` الحقيقي (channel / callback / semantic aliases / command / message) | **هذا بالضبط ما يحتاجه Event Simulation** — لا حاجة لإعادة تنفيذ منطق التوجيه؛ فقط تمرير raw update dict مصطنع لدالة حقيقية. المشكلة: الاسم يبدأ بـ `_` (private) |
| `bot.inspect()` → `BotSnapshot` | لقطة وصفية كاملة: commands، callbacks، events، middleware_count، has_error_handler، included_router_count، capabilities_available | **مطابق تماماً** لما طلبه المستخدم في "Inspector Integration" — لا حاجة لإعادة بناء هذه المعلومة، فقط استهلاكها |
| `titan.migration` | Knowledge API مصمم أصلاً ليُستهلك برمجياً من أدوات مثل Playground و Architect AI (موثق في ADR-007) | يؤكد أن Playground جزء من عائلة "أدوات تستهلك Core عبر API معلن"، لا امتداداً للـ Core نفسه |
| `Update` (`titan/update.py`) | يحوّل raw dict إلى واجهة قراءة (text, user_id, chat_id, callback_query, ...) | Event Simulation يحتاج فقط بناء raw dict صحيح الشكل؛ `Update` نفسه لا يحتاج تعديلاً |
| `Context.__init__` | يتطلب `update: Update` و `api: Telegram` حقيقيين (أو ما يوافق واجهتهما) | أي محاكاة حقيقية (لا mock) تحتاج `Telegram` بحالة لا تُجري شبكة فعلية — راجع القسم 4، التوتر 2 |
| `Router` | تسجيل معزول قابل للدمج عبر `bot.include()` | لا علاقة مباشرة بـ Playground v1 — قد تُستخدم لاحقاً لتحميل "مشروع قديم" (integration مع Migration Assistant، خارج v1) |

**ملاحظة مهمة:** لا يوجد اليوم أي مكوّن اسمه Playground أو ما يشبهه في الكود أو التوثيق. الإشارة الوحيدة هي سطر واحد في `docs/decisions/007-migration-assistant.md` يذكر Playground كمثال لأداة مستقبلية تستهلك `titan.migration` — لا تصميم، لا التزام.

## 3. هل المشكلة حقيقية؟

نعم، بثلاثة سياقات متوافقة مع نمط الأدوات السابقة (Inspector، Migration Assistant):

**أ) التعلّم:** مطور قادم من إطار آخر (PTB/aiogram) يحتاج أن "يرى" سلوك Titan قبل الالتزام به. `docs/migration/*.md` يشرح بالكلام؛ Playground يُري بالتنفيذ.

**ب) الـ debugging أثناء التطوير:** المطور الذي يملك بوتاً حقيقياً بالفعل يحتاج أحياناً عزل جزء واحد (middleware معين، handler معين) للتأكد من سلوكه دون تشغيل البوت كاملاً أو انتظار update حقيقي من Telegram.

**ج) المستقبل (Architect AI):** كما ذكر المستخدم، Architect AI يحتاج "runtime state" و"execution traces"، لا قراءة ملفات فقط. Playground هي الطبقة التي تنتج هذه المعلومة بشكل مُهيكل وقابل لإعادة الاستخدام برمجياً — بنفس منطق ADR-007 (طبقة توثيق بشرية + طبقة API برمجية).

**الحجم المتوقع لا يتجاوز الحاجة:** لا يوجد بديل أبسط (توثيق فقط لا يكفي، لأن "التجربة" هي جوهر الطلب لا الشرح).

## 4. أشكال التنفيذ الممكنة والتوترات المعمارية

### التوتر 1 — هل Playground يعتمد على mocks أم runtime حقيقي؟

المستخدم حسم هذا صراحة: **رؤيته أن يكون بيئة تشغيل حقيقية**، وليس محاكاة سطحية:

> إنشاء Context تجريبي، إرسال event وهمي، مشاهدة كيف يمر عبر النظام.

هذا يعني: Playground يجب أن يستخدم `bot._handle_update()` (أو واجهة عامة مكافئة تُبنى لهذا الغرض) وليس إعادة تنفيذ موازٍ لمنطق dispatch. إعادة التنفيذ تعني نسخة ثانية من منطق التوجيه، ستنحرف عن الأصل مع أول تعديل في `bot.py` — نفس الدرس المستفاد من ADR-006 (Inspector يقرأ من public API لا ينسخ الحالة).

**الخيارات:**
- **أ. Playground يستدعي `bot._handle_update()` مباشرة** — يعمل فوراً، لكنه يعتمد على method خاص (`_` prefix) غير مضمون في الـ CONTRACT. أي تغيير داخلي في التوقيع يكسر Playground بصمت.
- **ب. نضيف نقطة دخول عامة جديدة في Core** مخصصة لمعالجة raw update واحد بشكل معزول (مثلاً كإعادة تسمية/تعريض `_handle_update` بضمان استقرار). هذا "تعديل صغير في الـ Core" حسب `feature-workflow.md` (البند 4) — يحتاج تبريراً مستقلاً لأنه يزيد الـ frozen surface.
- **ج. Playground يبقى خارج حزمة `titan` تماماً (أداة/مشروع منفصل)** ويعتمد على الـ public API الحالي فقط (`bot.include()`, `bot.inspect()`) لبناء بوت حقيقي في الذاكرة ثم "حقن" update عبر استدعاء `_handle_update` من الخارج (Python لا يمنع الوصول لـ private، لكنه اتفاقية لا حماية).

**الأثر على شبكة Telegram الحقيقية:** بغض النظر عن الخيار، Playground لا يجب أن يجري أي HTTP call فعلي لـ Telegram. `Telegram` (في `telegram.py`) يبني على `aiohttp.ClientSession`. الحاجة هنا: نسخة من `Telegram` (أو interface بديل) تُسجّل الطلبات الصادرة (`send_message`, `answer_callback_query`, ...) بدلاً من إرسالها فعلياً — هذا "تسجيل" وليس "تزييف سلوك"، لأن الفرق الوحيد هو الوجهة (ذاكرة بدل شبكة)، لا المنطق.

### التوتر 2 — أين يسكن معمارياً؟

المستخدم اقترح صراحةً: `titan/playground/` كـ domain مستقل، أو أداة منفصلة "إذا كان أنسب معمارياً".

بالمقارنة مع السوابق:
- `titan.migration` — Core package لأنه Knowledge API يُستهلك برمجياً من داخل نفس البيئة التي تستخدم Titan.
- `titan.health` / `titan.inspector` — Core لأنهما يقرآن حالة بوت حي عبر public API.

Playground يختلف نوعياً: هو **بيئة تفاعلية للمطور نفسه**، ليست API تُستهلك من كود بوت حقيقي في الإنتاج. هذا يقترب أكثر من طبيعة أداة CLI/REPL مستقلة أكثر من كونه جزءاً من الـ frozen import surface في `CONTRACT.md`.

**السؤال المعماري الحقيقي:** هل Playground:
- **(أ)** حزمة داخل `src/titan/playground/` تُصدَّر بشكل صريح لكن **بدون** أن تدخل قائمة `# 1. Public API` المجمّدة في `CONTRACT.md` (مثل `titan.migration` اليوم)؟
- **(ب)** أداة CLI منفصلة (`titan-playground` أو ما شابه) تعتمد على `titan` كمكتبة فقط، تعيش في `tools/` أو حزمة منفصلة، ولا تدخل حزمة التوزيع الأساسية؟

الفرق عملي: (أ) يعني أي دالة عامة فيها تحتاج توثيقاً وقد تُعتبر جزءاً من "سطح مضمون" ضمنياً حتى لو لم تُذكر في CONTRACT.md صراحة (سابقة ADR-010 ناقشت هذا التمييز). (ب) يفصل تماماً ويسمح بالتجربة السريعة دون التزام.

### التوتر 3 — علاقته بـ Interactive Inspector تحديداً

المستخدم طلب أن يكون Playground "المستهلك الطبيعي" لـ Inspector. هذا **لا يحتاج أي تصميم إضافي** — `bot.inspect()` عام بالفعل ويُستدعى كما هو:

```python
snapshot = bot.inspect()
# Playground يعرض snapshot.commands, snapshot.events, ... كما هي
```

لا تعارض، لا تكرار. Playground طبقة عرض/تفاعل فوق بيانات Inspector الموجودة فعلاً.

### التوتر 4 — ما الحد الأدنى القابل للحياة (v1)؟

بناءً على حدود v1 التي حددها المستخدم (لا محرر كود، لا plugins، لا IDE، لا واجهة رسومية معقدة)، الحد الأدنى الوظيفي الذي يحقق "الهدف الأساسي" وحده هو **Event Simulation + Inspector Integration**:

1. بناء بوت `Titan` حقيقي في الذاكرة (المطور يسجل عليه handlers/middleware/routers تماماً كالمعتاد).
2. دالة واحدة تصنع raw update dict صحيح الشكل من مدخلات مبسطة (نص، chat_id، user_id، نوع الحدث) — بديل عن كتابة raw JSON يدوياً.
3. تمرير الـ update المُصنَّع إلى مسار المعالجة الحقيقي (سواء عبر تعريض `_handle_update` أو نقطة دخول عامة جديدة — قرار التوتر 1).
4. `Telegram` بديل (in-memory) يسجل كل استدعاء API بدل تنفيذه، ليراه المطور كـ "ماذا كان سيُرسل".
5. عرض `bot.inspect()` قبل وبعد التشغيل.

**خارج الحد الأدنى (مؤجل بوعي لِـ v1 القادمة أو أبعد):**
- Migration Assistant integration (تحميل مشروع قديم) — المستخدم نفسه وضعه كـ "ليس في v1 بالضرورة".
- Architect AI consumption (execution traces مُهيكلة للاستهلاك الآلي) — يحتاج تصميماً منفصلاً بعد استقرار v1 وبعد وجود Architect AI فعلياً كمستهلك حقيقي (نفس مبدأ رفض `by_tag()` في ADR-010: لا نبني API لمستهلك غير موجود بعد).
- أي واجهة رسومية — المستخدم استبعدها صراحة لـ v1.

## 5. الأسئلة المفتوحة للنقاش المعماري

1. **نقطة الدخول للـ Event Simulation** — هل نُعرِّض `_handle_update` (كسر التوصيف الخاص الحالي)، أم نضيف نقطة دخول عامة جديدة صريحة مخصصة لهذا الغرض؟ كلا الخيارين "تعديل صغير على الـ Core" حسب `feature-workflow.md` ويحتاج قراراً معمارياً مسبقاً قبل أي كود Playground.
2. **مكان Playground** — حزمة `titan.playground` غير مُصدَّرة في `CONTRACT.md` (شبيهة بـ `titan.migration`)، أم أداة/حزمة منفصلة تماماً خارج `src/titan/`؟
3. **الـ Telegram البديل (in-memory)** — هل يكون class جديد يطبّق نفس واجهة `Telegram` (Duck typing)، أم نحتاج تجريد صريح (Protocol/ABC) في `telegram.py` نفسه لضمان أن أي بديل مستقبلي (test doubles أخرى) يتبع نفس العقد؟ هذا قد يتقاطع مع اختبارات الوحدة الحالية التي قد تُنشئ بالفعل بدائل مشابهة — يحتاج فحصاً في `tests/` قبل القرار.
4. **شكل مُصنِّع الـ update (raw dict builder)** — هل هو جزء من Playground فقط، أم أداة عامة مفيدة أيضاً لكتابة اختبارات Titan نفسها (`tests/`) — ما قد يجعله موقعه الصحيح مساعد اختبار مشترك بدل حصره داخل Playground؟
5. **الاسم النهائي للحزمة/الأداة** — `titan.playground` كما اقترح المستخدم، أم اسم آخر يعكس طبيعتها كأداة لا Core module (تمييزاً عن `titan.migration`/`titan.health`)؟

## 6. تقييم أولي

**المشكلة حقيقية:** نعم — لا توجد اليوم طريقة لتجربة سلوك Titan دون بوت حقيقي متصل بـ Telegram.

**التداخل مع الأدوات الحالية:** لا تداخل — Playground يستهلك Inspector ولاحقاً Migration Assistant، لا يستبدل أياً منهما.

**الحجم المتوقع:** أكبر من Inspector (نموذج واحد + دالة) وأصغر من Migration Assistant (طبقتان كاملتان). يتطلب على الأقل: بديل `Telegram` in-memory، مُصنِّع raw update، وقرار معماري حول نقطة الدخول والموقع.

**التبعية الحرجة قبل أي كود:** حسم التوتر 1 (نقطة الدخول) والتوتر 2 (الموقع المعماري) — كلاهما يمس الـ Core الحالي أو حدود التوزيع، ويجب أن يُناقش صراحة قبل الكتابة، حسب تفضيل المستخدم الثابت "لا تُعاد صياغة المعمارية بدون نقاش".
