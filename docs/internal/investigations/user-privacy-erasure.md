# تحقيق — User Privacy & Erasure Protocol في Titan

**الحالة:** تحقيق أولي — تحديد نطاق. لا ADR. لا كود.

**المبدأ الفلسفي الذي يحكم هذا التحقيق:**

> نحن لا نقدم ميزة خصوصية فقط — بل نعالج نقص الخصوصية.

الفرق ليس لفظياً. ميزة الخصوصية تُضاف فوق مشكلة قائمة. معالجة النقص
تزيل سبب المشكلة قبل أن تنشأ. أي قرار في هذا التحقيق — أو في ADR
لاحقاً — يسأل نفسه أولاً:

> هل نحن نضيف طبقة جديدة، أم نزيل نقصاً كان موجوداً أصلاً؟

**السؤال الجوهري لهذا التحقيق:**

> لماذا احتفظنا بهذه البيانات أصلاً؟

ليس "كيف نحذفها" — بل ما المبرر لوجودها من الأساس.
الحذف يأتي لاحقاً، كنتيجة طبيعية لإجابة هذا السؤال.

## أولاً — ما يقوله المطورون الفعليون (لا الوثائق)

شكاوى متكررة من GitHub Issues وReddit وStack Overflow ومدونات المطورين
عن PTB وaiogram وtelebot. هذه ليست قراءة وثائق — هي أنماط ظهرت بعد
سنوات من الاستخدام الفعلي.

### النمط الأول — "لم أعرف أن الإطار يخزّن هذا"

الشكوى الأكثر تكراراً ليست "لا توجد أداة حذف" — بل "لم أكن أعلم أن
هناك شيئاً يُحذف أصلاً."

في PTB: `PicklePersistence` يكتب `user_data` كاملاً إلى الديسك. المطوّر
يضع كائناً مؤقتاً في `context.user_data` ظناً منه أنه ephemeral —
فإذا به يتحوّل إلى سجل دائم على الديسك بما فيه من PII لم يقصد
استمراره. **الإطار خزّن بالنيابة عنه بصمت.**

في aiogram: `MemoryStorage` — الافتراضي — يحتفظ بـ FSM state لكل مستخدم
تفاعل. المطوّرون يكتشفون بعد أشهر أن RAM تتضخم تدريجياً بـ user IDs
وبيانات حالة لمستخدمين أغلقوا البوت منذ أمد. لا يوجد TTL، لا يوجد
تنبيه، لا يوجد وثيقة تقول "هذا يتراكم."

**الجذر:** الإطار افترض أن التخزين خدمة — لم يسأل: هل هي ضرورية؟

### النمط الثاني — عدم التماثل بين التخزين والحذف

في كل مكتبة مدروسة: التخزين سطر واحد. الحذف يحتاج فهم internal schema.

المطوّرون الذين حاولوا بناء `/forgetme` أو امتثال GDPR وجدوا أنفسهم
يقرؤون كود المكتبة الداخلي لفهم أين تعيش البيانات — ثم يكتبون SQL
مباشراً أو يمسحون ملفات pickle يدوياً. لا API، لا خريطة.

**الجذر:** الإطار صمَّم التخزين، ونسي الحذف.

### النمط الثالث — بيانات "منسية" بدون مالك

في telebot وعدد من المشاريع المبنية على PTB: global dictionaries تتراكم
لأنها لا تملك دورة حياة. لا أحد يسأل "متى تنتهي حياة هذا المفتاح؟"
المفتاح `user_id → state` يُنشأ عند التفاعل الأول، ولا أحد يمحوه.

شكاوى من مستخدمي بوتات (ليس مطورين): عند حظر البوت أو إلغاء تشغيله،
لا يزول شيء من قاعدة بيانات البوت. المستخدم يعتقد أنه "غادر" — بينما
بياناته لا تزال تعيش في مكان لا يعرف أين.

**الجذر:** البيانات لا تملك صاحباً واضحاً ولا عمراً محدداً.

### النمط الرابع — opt-out بدل opt-in لميزات التخزين

كل المكتبات تخزّن بالافتراض وتتطلب جهداً لإيقاف التخزين — لا العكس.
لا توجد مكتبة واحدة قالت: "لا نخزّن شيئاً حتى تطلب منا ذلك صراحةً."

## ثانياً — أسئلة الجذر: لماذا احتفظنا؟

بعد مراجعة المكتبات الأخرى، نطرح نفس السؤال على Titan نفسها.

**تمييز أساسي قبل الأسئلة:**

Titan تدير نوعين من التخزين الدائم، وهما ليسا من نفس الجنس:

| النوع | المثال | من يملكه |
|---|---|---|
| **Protocol Storage** | `message_identity` في Links | هوية رسائل البوت — بروتوكول داخلي |
| **User Data** | `AskManager._pending` | حالة مرتبطة بمستخدم حقيقي |

الأسئلة التالية تُطرح على كل نوع بشكل منفصل — لأن الخلط بينهما هو
مصدر كثير من الالتباس في الوثيقة الأولى.

---

**هل Titan تجمع بيانات لا يحتاجها النظام الذي يستخدمها؟**

لا — `message_identity` يحتاجها نظام Links ليعمل: `titan_id` هو
الرابط الدائم نفسه. بدون السجل لا يوجد URL ثابت.

الصياغة الدقيقة: **Titan تجمع بيانات يحتاجها نظام Links — لكن ليس
كل بوت يحتاج نظام Links.** الذي لا يحتاجه هو البوت، وليس الميزة.
هذا الفارق مهم: المشكلة في نطاق التفعيل، لا في وجود التخزين.

**هل Titan لديها User Data تحتفظ بها أطول من اللازم؟**

نعم — `AskManager._pending` تحتفظ بـ `(chat_id, user_id) → Future`
بلا TTL. إذا لم يُكمل المستخدم flow — Future يبقى معلقاً إلى الأبد.

**هل يوجد شيء يجب أن يكون Opt-in؟**

Archive Layer (نص رسائل البوت) opt-in بالفعل: ✅ جيد.
Links نفسها تفعَّل في كل بوت — وهذا سؤال نطاق، لا سؤال خصوصية
مستخدم، لأن ما تخزّنه هو هوية رسائل البوت لا بيانات المستخدم.

**هل يستطيع المطوّر معرفة ما تخزّنه Titan تلقائياً؟**

لا — لا يوجد API لهذا. المطوّر لا يستطيع أن يسأل Titan:
"ماذا تحتفظ عن هذا المستخدم؟" — عليه قراءة الكود الداخلي.

## ثالثاً — جرد دقيق: ماذا تحتفظ Titan؟

### 3.1 بيانات عبور — تعيش update واحداً فقط

| المكان | البيانات | الحكم |
|---|---|---|
| `ctx` / `update.py` | user_id, chat_id, text, callback_data, raw JSON | ✅ ضرورية — تموت مع الـ update |
| `models/sender.py` | user_id, first_name, last_name, username | ✅ ضرورية — تموت مع الـ update |
| `models/chat.py` | chat_id, title, type, username | ✅ ضرورية — تموت مع الـ update |
| `models/message.py` | message_id, text | ✅ ضرورية — تموت مع الـ update |

هذه البيانات تمر عبر Titan كمياه في أنبوب — لا تتراكم، لا تُخزَّن.

### 3.2 ذاكرة Bot lifecycle — تعيش طالما البوت يعمل

| المكان | البيانات | ضرورية؟ | مشكلة |
|---|---|---|---|
| `bot.banned_users` | set[int] — user IDs | ✅ نعم — وظيفتها الاحتفاظ | — |
| `AskManager._pending` | `(chat_id, user_id) → Future` | ✅ نعم — للعمل الوظيفي | ⚠️ لا TTL — future معلّق إلى الأبد إذا لم يُرَد |

**مشكلة `AskManager`:** إذا بدأ مستخدم flow ولم يُكمله — Future يبقى
معلقاً حتى إعادة تشغيل البوت. لا يوجد طريقة لـ cleanup صريح. في بوت
به مئات المستخدمين النشطين، هذا تراكم حقيقي.

### 3.3 بيانات مستمرة — مُصنَّفة بحسب نوعها

التصنيف الأساسي قبل الجدول: **ليس كل تخزين دائم بيانات مستخدم.**

```
Permanent Resource Identity  ←  معرّفات موارد أنشأها البوت — دائمة بالتعريف
User Data                    ←  حالة مرتبطة بمستخدم حقيقي — لها lifecycle
```

**Schema الفعلي في `.titan/links.db`:**

```sql
-- Permanent Resource Identity — هوية رسائل البوت
-- titan_id لا يُحذف ولا يُعاد استخدامه أبداً
CREATE TABLE message_identity (
    titan_id              INTEGER PRIMARY KEY AUTOINCREMENT,
    bot_username          TEXT NOT NULL,
    chat_id               INTEGER NOT NULL,
    telegram_message_id   INTEGER NOT NULL,
    deleted               INTEGER NOT NULL DEFAULT 0,
    UNIQUE (chat_id, telegram_message_id)
);

-- Archive Layer — محتوى رسائل البوت (opt-in)
-- يُفعَّل بـ bot.links.enable_archive()
CREATE TABLE message_archive (
    titan_id    INTEGER PRIMARY KEY REFERENCES message_identity,
    text        TEXT,
    chat_type   TEXT NOT NULL DEFAULT 'unknown',
    sent_at     TEXT NOT NULL
);
```

| البيانات | التصنيف | ملاحظة |
|---|---|---|
| `titan_id` + `bot_username` + `chat_id` + `message_id` | **Permanent Resource Identity** | معرّف مورد — لا lifecycle، لا حذف، لا علاقة بـ `erase_user` |
| `deleted` flag | **حالة مورد** — ليس lifecycle | الرسالة كانت موجودة ثم حُذفت — هذه حقيقة تاريخية، لا مرحلة دورة حياة |
| نص الرسالة + وقت الإرسال | **Archive Layer — opt-in** | محتوى رسائل البوت |
| `user_id` | — لا يُخزَّن | ✅ قرار صحيح |

**فلسفة `titan_id` — مكتوبة في الكود:**

من `identity.py` مباشرةً:
> الهوية تصف لحظة وجود — الرسالة كانت، وهذه الحقيقة لا تُمحى.

من `store.py`:
> تعليم رسالة كمحذوفة مع الحفاظ على titan_id للأبد.

هذا ليس قيداً تقنياً — هو قرار معماري مقصود. `titan_id` أقرب إلى
commit hash في Git أو رابط رسالة Telegram من أنه "سجل قاعدة بيانات
يمكن حذفه." بمجرد وجوده، صار جزءاً من تاريخ النظام.

ما يتغيّر هو حالة المورد الذي يشير إليه الرابط:
- الرسالة قد تُحذف → `deleted = True`
- الأرشيف قد لا يكون موجوداً → لا سجل في `message_archive`

لكن `titan_id` نفسه خارج أي مفهوم حذف أو إعادة استخدام.

**ملاحظة لازمة:** `chat_id` في المحادثة الخاصة (private) يساوي رقمياً
`user_id` في Telegram — دوره في Links هو تحديد مكان الرسالة، لا
تعريف المستخدم. هذا التداخل حقيقي ويستحق توثيقاً، لكنه لا يحوّل
Permanent Resource Identity إلى User Data.

## رابعاً — مبادئ الخصوصية (Privacy Principles)

هذه مبادئ تُحكم قرارات Titan — تُكتب قبل أي API:

### تنبيه: الخصوصية ليست "لا تخزّن"

قبل المبادئ، تصحيح شائع: الخصوصية لا تعني عدم التخزين دائماً.
أحياناً التخزين ضروري وظيفياً — Links تحتاج `titan_id` لإنتاج URL ثابت.
المشكلة ليست في التخزين نفسه — بل في التخزين الذي يكون:
- **مجهولاً** — المطوّر لا يعرف أنه يحدث
- **غير محدود** — لا عمر، لا pruning
- **غير قابل للمحو** — أو قابل له وهمياً فقط (flag)

الخصوصية الجيدة: **تخزّن ما تحتاجه، تُعلن ما تخزّنه، تُمكِّن من محوه.**

### المبدأ الأول — أقل قدر من البيانات (Data Minimization)

Titan لا تجمع بيانات لا تؤدي وظيفة مباشرة وضرورية.

التطبيق: أي ميزة تتطلب تخزين بيانات تسأل أولاً: هل يمكن تقديم نفس
الوظيفة بدون هذه البيانات؟ إن كان الجواب نعم — لا تُخزَّن. إن كان
لا — تُخزَّن مع الإعلان الصريح وتوفير آلية المحو.

### المبدأ الثاني — كل تخزين دائم مبرَّر ومُعلَن وقابل للمحو

> كل تخزين دائم في Titan يجب أن يكون مبرَّراً ومُعلَناً وقابلاً للمحو،
> حتى لو كان افتراضياً.

هذا المبدأ يحل التناقض بين "Links تلقائية" و"Titan لا تخزّن بصمت":
Links تخزّن تلقائياً — لكن هذا التخزين مبرَّر (هوية الروابط)،
مُعلَن (هذه الوثيقة + التوثيق)، وله عقد دورة حياة واضح.

التطبيق:
- `message_identity`: مبرَّر ✅ — مُعلَن بحاجة إلى تحسين ⚠️ — عقد دورة الحياة يُحسم في ADR
- `AskManager._pending`: مبرَّر ✅ — مُعلَن ⚠️ — دورة حياة غائبة ⚠️

### المبدأ الثالث — الاحتفاظ الأقصر (Retention Minimization)

كل بيانات تملك عمراً محدداً. "إلى الأبد" ليست إجابة.

التطبيق:
- بيانات عبور: تموت مع الـ update ✅
- AskManager futures: تحتاج TTL أو آلية cleanup واضحة
- SQLite records: تحتاج حذفاً حقيقياً عند mark_deleted

### المبدأ الرابع — شفافية ما تحتفظ به Titan

المطوّر يستطيع دائماً معرفة ما تخزّنه Titan بالنيابة عنه.

التطبيق: لا يوجد API لهذا حالياً. وجوده شرط للامتثال الحقيقي.

### المبدأ الخامس — لكل تخزين حذف مقابل

Titan لا تضع آلية تخزين دون أن تضع بجانبها آلية حذف حقيقي.

التطبيق: `mark_deleted` (flag فقط) لا يُعدّ حذفاً بهذا المعنى.

### المبدأ السادس — المستخدم النهائي له حق المحو

مستخدم البوت — ليس المطوّر فقط — يحق له طلب محو بياناته.

Titan لا تُنفّذ هذا بنفسها (لا تعرف من "المستخدم النهائي")، لكنها
توفر API يُمكِّن المطوّر من الامتثال لهذا الطلب بسطر واحد.

**هذا المبدأ هو ما يُفرّق بين "ميزة خصوصية" و"معالجة نقص الخصوصية."
الأطر الأخرى تجعل المطوّر يكافح لبناء `/forgetme` — Titan يجب أن
تجعله بديهياً.**

### المبدأ السابع — لا بيانات "منسية"

كل بيانات تملك صاحباً ودورة حياة موثّقة. لا يوجد شيء يُخزَّن ثم
"يُنسى."

### المبدأ التاسع — الخصوصية قابلة للتركيب (Privacy Composability)

> كل Module يدخل Titan ويُنشئ تخزيناً دائماً يجب أن يُعلن عقد دورة
> حياة بياناته.

هذا المبدأ يمنع مشكلة "نسينا أن X كان يخزّن Y" بعد سنتين.

**ما يُعلنه كل Module:**
- ماذا يخزّن ولماذا
- هل هو Protocol Storage أم User Data
- كيف يُمحى (User Data) أو كيف تنتهي دورة حياته (Protocol Storage)
- كيف يظهر في `data_held_for` إن كان User Data

**التطبيق:** Module لا يُطبّق هذا العقد يُعدّ ناقصاً من منظور
المراجعة الهندسية — ليس مرفوضاً بالضرورة، لكنه يحمل ديناً معمارياً
صريحاً حتى يُكتمل.

في الوثائق الهندسية (`module_spec.md`): يتحوّل هذا إلى checklist
قابل للتحقق في مراجعة الـ Pull Request.

### المبدأ الثامن — الخصوصية لا تُفاجئ ولا تُثقل

> الخصوصية لا ينبغي أن تفرض عبئاً على المطور، ولا أن تفاجئه.

وجهان لنفس المبدأ:

**لا مفاجأة:** Titan لا تخزّن شيئاً بصمت. كل ما تحتفظ به Titan يُوثَّق
صراحةً ويمكن الاطلاع عليه برمجياً. المطوّر لا يكتشف وجود `.titan/links.db`
بعد ستة أشهر من الإنتاج.

**لا عبء:** المطوّر لا يكتب عشرين سطراً ليحترم خصوصية المستخدم.
`/forgetme` يجب أن يكون سطراً واحداً — لا schema diving، لا SQL مباشر،
لا قراءة كود داخلي. الخصوصية لا تُلاحظ في الاستخدام العادي، وتكون
في متناول اليد عند الحاجة.

هذا المبدأ هو الجسر بين الفلسفة والـ API: إذا كان `erase_user` يتطلب
معرفة بالتخزين الداخلي — فالمبدأ لم يُطبَّق.

## خامساً — ما الذي تتحمله Titan وما الذي لا تتحمله

### تتحمله Titan

| | التفصيل |
|---|---|
| **Permanent Resource Identity** (Links): إعلان ما هو وسبب ديمومته | Titan تنشئ معرّفات دائمة — توفر وضوحاً عن طبيعتها لا عن حذفها |
| **User Data** (AskManager): توفير TTL أو آلية cleanup | Titan تديرها — Titan مسؤولة عن منع التراكم |
| **User Data**: توفير `erase_user` كعقد محو عام | كل Module يخزّن User Data يُسجَّل تلقائياً ضمن هذا العقد |
| شفافية ما تخزّنه — `data_held_for` | المطوّر يستطيع معرفة ما تحتفظ به Titan ببيانات المستخدم |

### لا تتحمله Titan

| | السبب |
|---|---|
| بيانات كود المطوّر (قواعد بياناته الخاصة) | قراره تماماً |
| Telegram's servers | خارج نطاق أي إطار عمل |
| الامتثال القانوني للمطوّر (GDPR/PDPL) | Titan ليست مستشاراً قانونياً |
| `ctx.raw` / `model.raw` الذي يقرر المطوّر تخزينه | استخدم الـ escape hatch بوعي |
| بيانات الإدارة (user_id في `banned_users`) | المطوّر وضعها — المطوّر يحذفها |

**خط الفصل الواضح:**
```
ما تجمعه وتخزّنه Titan تلقائياً  →  مسؤولية Titan
ما يجمعه المطوّر باستخدام Titan  →  مسؤولية المطوّر
```

## سادساً — واجهة عامة مقترحة (بعد المبادئ)

هذا القسم يأتي بعد المبادئ عمداً — الـ API نتيجة، لا نقطة بداية.

### المقترح الأساسي

```python
# من منظور المستخدم النهائي — المطوّر يستدعيه عند طلب "احذف بياناتي"
await bot.erase_user(user_id=123456789)

# تقرير ما تحتفظ به Titan عن مستخدم — للشفافية
report = bot.data_held_for(user_id=123456789)
# → {"links_records": 47, "pending_asks": 0}
```

**الواجهة تفكّر بالمستخدم — التنفيذ يترجم داخلياً:**
المطوّر لا يفكّر بـ `chat_id`. يفكّر بـ "المستخدم الذي طلب حذف بياناته."
اليوم يعيش هذا التخزين في `chat_id` — لأن Links تخزّن بمنظور الدردشة.
غداً قد يُضاف `user_id` صريح. بعد عامين قد تكون هناك Userbot Support.

الـ API لا تتغيّر. التنفيذ الداخلي يترجم `user_id` إلى ما يناسب كل
storage في وقته — اليوم إلى `chat_id` للمحادثات الخاصة، لاحقاً لأي
شيء آخر. هذا شأن التنفيذ، لا شأن الواجهة.

### `erase_user` — عقد عام، لا وصف لتطبيق

`erase_user` ليس API مرتبطاً بـ AskManager أو بأي module محدد.
هو **عقد**: "احذف كل ما تحتفظ به Titan عن هذا المستخدم في نطاق User Data."

اليوم: `AskManager` هو المكوّن الوحيد الذي يخزّن User Data — إذن
`erase_user` يلغي pending asks لهذا المستخدم.

غداً: أي Module جديد يُعلن أنه يخزّن User State (Preferences،
Sessions، Rate Limits، Profiles...) يُسجَّل تلقائياً ضمن هذا العقد —
ويصبح `erase_user` مسؤولاً عنه دون تغيير في الـ API.

**ما لا يلمسه `erase_user` أبداً:**
- `message_identity` و`message_archive` — Protocol Storage، لا User Data
- `bot.banned_users` — المطوّر وضعها، المطوّر يحذفها

### التنفيذ الحالي (اليوم فقط)

1. يلغي أي pending asks لهذا المستخدم في `AskManager`
2. _(لا شيء آخر — لأنه لا يوجد User Data آخر اليوم)_

### ما لا يفعله

- لا يمس `bot.banned_users` (المطوّر يديرها بنفسه)
- لا يرسل أي شيء لـ Telegram
- لا يمس بيانات المطوّر الخاصة

## سابعاً — الأسئلة المعمارية الحقيقية

### السؤال الأول — هل تعتبر Titan نفسها مسؤولة عن دورة حياة البيانات التي أنشأتها؟ (الأهم)

هذا هو السؤال الجوهري الذي يسبق كل قرار آخر في هذا النطاق.

**إذا كان الجواب نعم:**
- `erase_user` ليس اختياراً — بل نتيجة مباشرة
- `data_held_for` متوقَّع، لا ترف
- كل Module جديد يخزّن User Data يلتزم بعقد دورة الحياة
- `/mydata` و`/forgetme` أوامر محجوزة في كل بوت Titan
- ADR لا يناقش "هل نُوفِّر erase_user؟" — بل "كيف نُطبِّقه؟"

**إذا كان الجواب لا:**
- المسؤولية تنتقل للمطوّر كاملاً
- Titan تُوفِّر أدوات فقط، لا ضمانات
- نصف هذه الوثيقة يسقط

**الموقف:** الجواب **نعم** — وبحدود واضحة:

> Titan مسؤولة عن دورة حياة البيانات التي تُنشئها وتُديرها داخلياً
> بالنيابة عن المطوّر. هذه المسؤولية لا تمتد لبيانات المطوّر الخاصة.

الحدود مكتوبة في القسم الخامس بالتفصيل. هذا الموقف يجب أن يكون
**أول قرار في ADR-001** — لأن كل قرار لاحق يشتق منه.

### السؤال الثاني — ما هو نطاق تفعيل Links؟

Links تُفعَّل في كل بوت Titan (`bot.py` السطر 119). ما يُخزَّن هو
Protocol Storage — هوية رسائل البوت — لا User Data. لذلك هذا ليس
سؤال خصوصية مستخدم، بل سؤال نطاق بروتوكول: هل كل بوت يحتاج Links؟

**ثلاثة خيارات:**
- **(أ) Links تبقى تلقائية** — السلوك الحالي. كل بوت يملك protocol storage منذ اليوم الأول.
- **(ب) Links opt-in صريح** — `bot.links.enable()`. من لا يحتاج `/link` لا يُنشأ له DB.
- **(ج) Lazy registration** — لا تسجيل عند الإرسال، بل عند أول استدعاء فعلي لـ `/link`. التخزين يبدأ عند الحاجة الفعلية.

**يُحسم في ADR — ليس افتراضاً.**

### السؤال الثاني — أين تعيش `erase_user`؟

Core لا تعرف عن Links ولا عن AskManager. هل:
- (أ) `bot.links.erase_user(user_id)` و`ask.erase_user(user_id)` مستقلتان — المطوّر يستدعي كلاً منهما
- (ب) نظام registration: كل مكوّن يُسجَّل كـ "data holder" — `bot.erase_user` تُنادي الجميع
- (ج) `bot.erase_user` تعمل فقط على ما تعرفه Core، والباقي على المطوّر

**الجواب يحدد شكل الـ API. يُحسم في ADR.**

### السؤال الثالث — `data_held_for` أم لا؟

هل نحتاج API للشفافية في v1، أم نقتصر على المحو ونؤجل الـ report؟

المبدأ الرابع يتطلبه — لكن تعقيده أعلى من `erase_user`. يمكن تأجيله
لـ v1.1 إذا قرّرنا أن المحو الحقيقي يكفي كخطوة أولى.

### السؤال الرابع — AskManager TTL

TTL بدون آلية cleanup يعني بيانات منسية. الخيارات:
- (أ) TTL بـ `asyncio.sleep` — يُعقّد AskManager
- (ب) cleanup عند `erase_user` فقط — بدون TTL تلقائي
- (ج) توثيق واضح بأن dangling futures خطر، والمطوّر مسؤول

## ثامناً — ما هو إلزامي وما هو اختياري للمطوّر

هذا القسم يجيب على سؤال حقيقي: أين تقع مسؤولية Titan وأين تقع
مسؤولية المطوّر — وأين تُلزم Titan المطوّر بشيء لا يستطيع تجاهله؟

**الخلفية:** ليس كل المطورين يعطون الأولوية لخصوصية مستخدميهم.
بعضهم يتجاهلها. بعضهم لا يعرف. إطار العمل الذي يترك كل شيء
للمطوّر يحمّل المستخدم النهائي ثمن هذا القصور. Titan يجب أن تُحدد
ماذا تفرض وماذا تترك.

### إلزامي — Titan تُلزم به المطوّر

| الإلزام | المبرر |
|---|---|
| **`erase_user` موجودة دائماً كـ API** | Titan تخزّن بالنيابة عن المطوّر — يجب أن يُمكَّن من المحو حتى لو لم يطلبه |
| **التوثيق الصريح لكل ما تخزّنه Titan** | المطوّر لا يحق له الجهل بما يجري بالنيابة عنه |
| **الحذف الحقيقي — لا flags** | `mark_deleted = 1` ليس حذفاً. Titan لا تقدّم وهم الحذف |

**ملاحظة مهمة:** "إلزامي" لا يعني تشغيل كود تلقائياً — يعني أن Titan
تضمن أن الأداة موجودة وأن الوثيقة تقول أين تعيش البيانات. المطوّر
الذي يتجاهل `erase_user` بعد ذلك — هذا قراره. لكن Titan لا تجعله
ممكناً بالجهل.

### اختياري — المطوّر يقرر

| الاختياري | التفصيل |
|---|---|
| **استدعاء `erase_user`** | Titan توفر الأداة — المطوّر يقرر متى يستدعيها |
| **تفعيل Archive Layer** | نص الرسالة لا يُخزَّن إلا بقرار صريح — opt-in حقيقي |
| **بناء `/forgetme` command** | Titan توفر `erase_user`، لكن command المستخدم هو مسؤولية المطوّر |
| **كيفية إبلاغ المستخدم** | Titan لا تتدخل في تجربة المستخدم النهائي |
| **أي تخزين يبنيه المطوّر بنفسه** | قواعد البيانات الخاصة، الـ logs، الـ caches — ليست شأن Titan |

### الحد الفاصل: أين تنتهي Titan؟

```
Titan مسؤولة عن ←  جعل الصواب سهلاً والخطأ واضحاً
Titan ليست مسؤولة عن ←  إجبار المطوّر على فعل الصواب
```

الفرق بين "الصواب سهل" و"الإجبار على الصواب" هو فرق فلسفي لـ Titan:
Titan تُقدّم `erase_user` بسطر واحد وتوثيق واضح — لا تمنع البوت من
العمل إذا لم يُنفَّذ. هذا خط مناسب لإطار عمل لا لمنصة.

## عاشراً — الخطوات التالية لفتح ADR

**ADR-001 يبدأ بقرار واحد — قبل أي تفصيل:**

> هل تعتبر Titan نفسها مسؤولة عن دورة حياة البيانات التي أنشأتها
> وتُديرها داخلياً بالنيابة عن المطوّر؟

الجواب "نعم" يُغلق نصف القرارات اللاحقة تلقائياً.

---

**بعد حسم ADR-001، بالترتيب:**

1. **نطاق Links** — تبقى تلقائية أم lazy registration أم opt-in؟
   هذا سؤال بروتوكول، لا خصوصية مستخدم. يؤثر على حجم Protocol Storage.

2. **موقع `erase_user`** — registration pattern أم core مباشر؟
   يحدد كيف يُعلن كل Module نفسه ضمن User Data Lifecycle.

3. **`/mydata` و`/forgetme`** — أوامر محجوزة في كل بوت Titan.
   Titan تملك السلوك، المطوّر لا يستطيع override أو disable.

4. **`data_held_for`** — مستويان: Developer report و Human-readable.
   يُحسم شكله بعد حسم موقع `erase_user`.

5. **AskManager TTL** — آلية cleanup للـ pending futures.

---

**ما لا يحتاج ADR:**

- `message_identity` ليس User Data، وليس "Storage بدورة حياة" —
  هو Permanent Resource Identity. `titan_id` لا يُحذف ولا يُعاد
  استخدامه. ما يتغيّر هو حالة المورد (`deleted`)، لا المعرّف نفسه.
  هذا قرار معماري مكتوب في الكود، لا في هذه الوثيقة.
- Archive Layer محسوم: opt-in، محتوى رسائل البوت، مستقل عن `erase_user`.
