# تحقيق — Userbot Support في Titan

> **⚠️ قرار تأجيل واعٍ — 2026-07-15**
>
> Userbot Support مؤجَّلة بقرار صريح. هذا ليس رفضاً — هو توقيت.
>
> **لا يُستكمل** أي كود أو تجريد أو adapter أو تكامل MTProto أو
> تحضير معماري متعلق بـ Userbot حتى يتحقق أحد الشروط التالية:
> 1. حاجة عملية حقيقية داخل Titan نفسها.
> 2. طلب واضح من المستخدمين أو المساهمين.
> 3. أصبح هدفاً مخططاً صريحاً لـ Titan v2.
>
> التركيز يعود الآن لـ **Titan Core** والخارطة الحالية.
> انظر قسم "مؤجلة بوعي" في `ROADMAP.md` للسجل الرسمي.

---

**الحالة:** اتجاه معماري مبدئي مختار — Pending ADR — **الخيار (هـ)
Unified Telegram Surface**. هذا ليس Architecture Decision نهائياً.
لا ADR بعد. لا كود بعد.

**الهدف:** فهم حجم وطبيعة التغيير المطلوب لدعم حسابات Userbot-style
(MTProto) إلى جانب Bot API الحالية، وتحديد السؤال المعماري الحقيقي
قبل أي التزام بالتنفيذ.

**لا تنفيذ في هذه المرحلة. لا ADR في هذه المرحلة.** هذا الملف تحقيق
وتصميم تمهيدي فقط — القرار المبدئي أدناه يحدد *الاتجاه*، لا التنفيذ.

---

## السؤال الحقيقي — ليس "هل نضيف adapter؟"

الصياغة الأولى للفكرة في `ROADMAP.md` (#10) كانت: *"دعم سطحَي تفاعل
Telegram: Bot API + Userbot-style accounts."* هذه صياغة مضلِّلة —
توحي بأن المطلوب هو adapter إضافي بجانب `TelegramAdapter` الحالي،
بنفس الحجم الذي أُضيف به دعم أي API method جديدة.

هذا ليس صحيحاً. Bot API وMTProto ليسا "نفس الشيء بواجهتين مختلفتين" —
هما بروتوكولان مختلفان جذرياً، بنموذجي هوية مختلفين، وبنموذجي أحداث
مختلفين تماماً. أي محاولة لدعمهما معاً تُجبر Titan على الإجابة عن سؤال
لم يُطرح صراحةً من قبل.

**تحديث الصياغة (2026-07-12):** الصياغة الأولى لهذا السؤال كانت
"هل Titan إطار Bot API أم Userbot؟" — وهي صياغة ثنائية مضلِّلة توحي
بأن الاختيار هو بين هوية واحدة أو أخرى. الصياغة الأدق، بعد مراجعة
معمارية، هي:

> هل Titan إطار **لمعالجة Telegram interactions بغض النظر عن سطح
> الاتصال**؟

الفرق ليس لفظياً. الصياغة الأولى تقود إلى تفكير "إما/أو" (اختر هوية
واحدة). الصياغة الثانية تفتح مساحة لتصوّر ثالث: أن يبقى Titan إطاراً
واحداً بمفهوم معالجة موحّد (Router → Middleware → Context → Handlers)
تغذّيه أكثر من قناة اتصال، دون أن يتطابق سطحاها بالضرورة في كل قدرة.
هذا التصور مفصَّل في **الخيار (هـ)** أدناه.

**تصحيح ثانٍ على الصياغة (2026-07-12) — مراجعة نقدية خارجية:** حتى
الصياغة أعلاه لا تزال تضع السؤال المركزي عند "كيف نجعل `Update`
موحّداً؟" — وهذا سؤال خاطئ من الأساس، لأنه يفترض ضمنياً أن `Update`
هي المفهوم الصحيح لمجرد أنها موجودة اليوم. **وجود ملف قديم ليس سبباً
معمارياً.** السؤال الأصوب الذي يجب أن يقود التصميم فعلياً هو:

> ما هو **Canonical Interaction Model** الذي يمرّ داخل Titan Core،
> بغض النظر عن اسم الصنف الذي يحمله اليوم؟

`titan.update.Update` قد يكون — أو لا يكون — الاسم الصحيح لهذا
النموذج بعد الفصل. القرار بين "الإبقاء على `Update` بعد تفريغها من
منطق Bot API" و"استخراج `titan.events` بأنواع منفصلة
(`MessageEvent`/`CallbackEvent`/`MemberEvent`) وجعل `Update` غلافاً/
حاوية دورة حياة (`Event → Update → Context`) فوقها" — هذا **قرار
تصميم يُحسم عند كتابة الـ ADR بناءً على ما يخدم النموذج الكانوني
فعلياً، لا بحكم الحفاظ على اسم موجود مسبقاً.**

تيليجرام نفسها توفر الطريقين. لكن أطر العمل الأخرى (Pyrogram، Telethon،
aiogram، PTB) اختارت طريقاً واحداً وتخصصت فيه. Titan حتى الآن اختارت
Bot API ضمنياً — لم يُعلَن هذا كقرار واعٍ في CONTRACT.md، لكنه القرار
الفعلي بحكم كل سطر كود موجود.

**هذا أكبر قرار معماري منذ إنشاء Titan نفسها** — ليس لأنه صعب تقنياً
فقط، بل لأنه يُعيد تعريف نطاق المشروع.

## الفرق الجذري بين Bot API وMTProto

| المحور | Bot API (الحالي) | MTProto (Userbot) |
|---|---|---|
| **البروتوكول** | HTTP REST — طلبات/ردود JSON بسيطة | بروتوكول ثنائي مخصص (MTProto)، تشفير طبقة التطبيق، اتصال دائم |
| **النقل** | `aiohttp` فوق HTTPS — مكتبة قياسية | لا يوجد HTTP هنا إطلاقاً — يحتاج مكتبة MTProto كاملة (لا يوجد تطبيق خفيف داخل aiohttp) |
| **الهوية** | Bot Token من BotFather | رقم هاتف + `api_id`/`api_hash` من my.telegram.org + جلسة مصادقة (session file) قابلة للسرقة إذا سُرِّبت |
| **استقبال الأحداث** | Long polling عبر `getUpdates(offset)` — نموذج pull بترتيب واضح | اتصال TCP دائم، السيرفر يدفع التحديثات (`Updates` push) بلا offset — نموذج مختلف تماماً في ترتيب المعالجة والـ backoff |
| **شكل الحدث** | JSON مسطح: `message`, `callback_query`, `channel_post` | بنية MTProto TL (Type Language): `UpdateNewMessage`, `UpdateShortMessage`, `UpdateNewChannelMessage`، عشرات الأنواع الفرعية |
| **القدرات** | Inline keyboards، `answerCallbackQuery`، `setMyCommands` — مفاهيم بوت رسمية | لا callback_query بنفس المعنى (userbot لا يُنشئ لوحات inline بنفس القيود)، لا commands menu، لكن قدرات مختلفة كلياً: قراءة سجل الدردشة الكامل، الانضمام كمستخدم عادي، إرسال من حساب شخصي |
| **القيود القانونية/السياسية** | مُعتمد رسمياً من Telegram كنموذج بوت | حسابات MTProto (Userbot) مدعومة رسمياً ضمن Telegram API نفسه — لكن بعض أنماط الأتمتة المبنية عليها قد تخالف [شروط استخدام Telegram](https://core.telegram.org/api/terms) أو سياسات مكافحة إساءة الاستخدام؛ المشكلة في نمط الاستخدام، لا في سطح MTProto بحد ذاته |

**النتيجة:** لا يوجد "قاسم مشترك تقني" بين الاثنين على مستوى النقل
(transport). القاسم المشترك الوحيد الممكن هو **مفهوم `Update` الموحّد
بعد الترجمة** — أي أن يترجم كل backend حدثه الخام إلى نفس الشكل الذي
يفهمه `titan.update.Update` أو ما يعادله.

## ما الذي بُني بالفعل ويُسهّل هذا (نقاط قوة حقيقية)

هذا الجزء مهم — Titan ليست في نقطة الصفر:

1. **`feed_update()` (ADR-011, Playground)** — مدخل رسمي لتغذية update
   من أي مصدر، غير مرتبط بـ `getUpdates` polling. صُمِّم صراحةً مع ذكر
   Userbot Support كأحد المستفيدين المستقبليين. هذا يعني أن **حلقة
   المعالجة الداخلية (`_handle_update`) منفصلة فعلاً عن آلية الجلب**.

2. **فصل `Telegram` (transport) عن `Context`/`Router`/`Middleware`
   (منطق التوجيه)** — منطق الـ routing لا يعرف شيئاً عن HTTP أو
   `aiohttp`. هذا يقلل نظرياً من حجم التغيير المطلوب في `bot.py`
   نفسها.

3. **`Update` هي طبقة ترجمة صريحة بالفعل** — `titan.update.Update`
   لا تُمرِّر JSON الخام للـ Context مباشرة، بل تستخرج حقولاً موحّدة
   (`text`, `chat_id`, `user_id`, ...). من حيث المبدأ، لا شيء يمنع
   وجود دالة ترجمة موازية: `Update.from_mtproto(raw_mtproto_event)`.

4. **مبدأ "Explicit over implicit" الموثَّق في `design_notes.md`**
   — "This principle is implementation-independent. It would apply
   equally under webhook support, Userbot mode, or a larger
   ecosystem." — الفلسفة العامة لم تُبنَ لتفترض Bot API حصراً، حتى
   لو كان الكود يفترض ذلك.

## نقاط الالتصاق الفعلية بـ Bot API (حيث سيحدث الاحتكاك الحقيقي)

هذه ليست تفاصيل تنفيذ تافهة — هي المكان الذي سيتحدد فيه حجم القرار:

### 1. `titan.telegram.Telegram` — بوابة النقل الوحيدة

كل طلب في Titan يمر عبر `Telegram.request()` التي تفترض:
- `base_url = https://api.telegram.org/bot{token}`
- طلب HTTP POST بجسم JSON، رد JSON بحقل `ok`

لا يوجد أي تجريد (`abstract transport`) هنا — `bot.py`، `ctx.py`،
و`adapter.py` كلها تحمل مرجعاً مباشراً لكائن `Telegram` (`self._api`).
دعم MTProto يعني إما:
- (أ) واجهة transport مجردة (`TelegramTransport` protocol) يُطبِّقها
  كل من `Telegram` (Bot API) وbackend جديد (MTProto)، أو
- (ب) كائن Telegram مختلف كلياً يُمرَّر لـ `Context`، بشرط أن يُطبِّق
  نفس الطرق (`send_message`, `get_me`, ...) — duck typing بدل abstract
  base، وهذا هو الأسلوب الذي اتبعه `titan.playground.RecordingTelegram`
  بالفعل (بديل duck-typed لـ `Telegram` لأغراض الاختبار).

نقطة (ب) مهمة: **`RecordingTelegram` في Playground هو دليل حي أن
Titan يمكنها العمل مع أي كائن يطبّق نفس الواجهة، لا `Telegram` تحديداً.**
هذا يقلل من حجم المخاوف حول (أ) — لكنه لا يحل مشكلة أن معظم methods في
`Telegram`/`TelegramAdapter` تحمل أسماء ومعاملات Bot-API-specific
(`answer_callback_query`, `set_my_commands`) لا معنى مباشراً لها في
MTProto (حتى لو كان لبعضها مكافئ بمعاملات مختلفة تماماً).

### 2. حلقة `run_async()` — polling بـ offset

`run_async()` مبنية بالكامل حول:
```python
updates = await self._api.get_updates(offset=self.offset + 1)
```
هذا نموذج pull مع offset متسلسل. أي MTProto client (Pyrogram/Telethon
أو تطبيق خام) يعمل بنموذج **push** فوق اتصال دائم — لا offset، لا
"استئناف من حيث توقفت" بنفس الطريقة (له آلية استئناف خاصة به على
مستوى الجلسة). دعم Userbot الحقيقي يعني أن `run_async()` بشكلها
الحالي **لن تُستخدَم إطلاقاً** لهذا الـ backend — سيحتاج مسار تشغيل
مختلف بالكامل، مع أن كلاهما ينتهي عند `feed_update()`/`_handle_update()`
نفس النقطة (وهذا مطمئن جزئياً بحكم البند السابق).

### 3. الاعتماد على مكتبة خارجية — تعارض فلسفي محتمل

Titan لا تعتمد على أي شيء غير `aiohttp` (انظر `pyproject.toml`).
تطبيق MTProto من الصفر مشروع ضخم مستقل بحد ذاته (تشفير، مفاوضة مفاتيح،
إدارة جلسة) — لا أحد يبني هذا داخل framework توجيه بسيط. الخيار
الواقعي الوحيد هو الاعتماد على مكتبة ناضجة مثل Pyrogram أو Telethon
كـ dependency اختيارية (`titan[userbot]` extra).

هذا يعني: **Titan لن "تدعم" MTProto فعلياً — ستكتب طبقة ترجمة/تكامل
فوق مكتبة أخرى تدعمه.** هذا نمط شائع (aiogram وPTB لا يطبّقان HTTP من
الصفر أيضاً، يعتمدان على `aiohttp`/`httpx`) لكنه يستحق أن يُقال بصراحة:
لن يكون "Titan توفر MTProto"، بل "Titan تُوفِّر واجهة موحّدة فوق مكتبة
MTProto موجودة." هذا التزام صيانة طويل الأمد (متابعة تغييرات تلك
المكتبة) لم يكن موجوداً من قبل.

**تنبيه مهم — هذا ليس سؤال "أي مكتبة" بل سؤال نطاق ملكية أسبق منه.**
المقارنة مع aiogram/PTB أعلاه غير دقيقة كاملةً: تلك الأطر تعتمد على
`aiohttp`/`httpx` لنقل *بايتات HTTP الخام* فقط — وهي بنية تحتية عامة
لا علاقة لها بـ Telegram أو بفلسفة أي framework. الاعتماد على
Pyrogram/Telethon مختلف جوهرياً: هذه المكتبات لا تنقل بايتات خام، بل
**تُطبِّق بروتوكول Telegram نفسه واجهاته وأشكال بياناته** — وهذا هو
تحديداً ما تملكه Titan بنفسها في جانب Bot API (`titan.telegram.Telegram`,
`titan.adapter.TelegramAdapter`). فاختيار مكتبة MTProto خارجية ليس
مجرد "استبدال aiohttp بمكتبة أنضج" — بل تنازل عن ملكية طبقة الاتصال
بالبروتوكول نفسها لصالح كود خارجي، في حين تحتفظ Titan بملكية هذه
الطبقة بالكامل على جانب Bot API. هذا يخالف مبدأً أساسياً في هوية
Titan: الاستقلالية ليست فقط في `Router`/`Context`/بنية Core، بل في
امتلاك الطبقات الأساسية لبروتوكول الاتصال نفسه.

**السؤال الحقيقي الذي يجب حسمه *قبل* اختيار أي مكتبة:**
هل MTProto surface — إن بُني — هو **سطح تملكه Titan رسمياً** (بمعنى:
تطبيق بروتوكول MTProto ذاته ضمن حدود المشروع، حتى لو استعان بمكتبة
تشفير/نقل منخفضة المستوى فقط)، أم **تكامل خارجي** (Titan تكتب طبقة
ترجمة فوق Pyrogram/Telethon الجاهزتين، دون أن "تملك" بروتوكول
MTProto نفسه)؟

الجوابان يقودان لنتائج مختلفة جذرياً:
- إن كان Userbot Support جزءاً من **هوية Titan نفسها** (بمعنى: Titan
  "تتكلم" MTProto كما تتكلم Bot API) → النتيجة المنطقية هي أن Titan
  تحتاج لامتلاك تطبيق MTProto الأساسي بنفسها — مشروع أكبر بكثير من
  مجرد طبقة ترجمة، ويقترب من حجم مشروع مستقل قائم بذاته.
- إن كان الهدف فقط "توحيد المعالجة فوق أي سطح Telegram متاح"، دون
  الحاجة لامتلاك بروتوكول الاتصال ذاته → الاعتماد على مكتبة خارجية
  مقبول فلسفياً، والخيار (هـ) يبقى كما هو دون تناقض.

**هذا سؤال نطاق مسؤولية (scope of ownership)، لا سؤال اختيار مكتبة —
ويستحق قراراً معمارياً مستقلاً بذاته (في الـ ADR نفسه أو قبله)، منفصلاً
عن تصميم طبقة الترجمة الموصوف في القسم التالي. اختيار Pyrogram أو
Telethon أو أي مكتبة أخرى قبل حسم هذا السؤال يحل المشكلة التقنية
اليوم، لكنه قد يخلق التزاماً فلسفياً متناقضاً مع هوية Titan لاحقاً —
دون أن يكون هناك قرار واعٍ اتُّخذ بهذا الخصوص.**

### 3ب. رأي هندسي — أين ترسم Titan خط الملكية عملياً (2026-07-12)

هذا رأي هندسي صريح كطرف في القرار، لا حسم نهائي — القرار يبقى بيد
المشروع، ولا يزال هذا توثيقاً بلا كود وبلا ADR.

**أتفق مع المبدأ، وأختلف مع تعريف "الملكية" الحرفي الكامل.** امتلاك
Titan لسطح MTProto منطقي ومتّسق مع هويتها — **لكن يجب التمييز بدقة بين
نوعين مختلفين جداً من "الملكية":**

1. **ملكية السطح (surface ownership)** — امتلاك: شكل الـ API الذي
   يراه مستخدم Titan (`bot.userbot.*` أو ما يعادله)، سياسة الجلسات
   ودورة حياتها، ترجمة الأحداث الواردة إلى `Update`/النموذج الداخلي،
   معالجة الأخطاء والتكامل مع `feed_update()`/`Router`/`Middleware`.
   هذا **يجب** أن تملكه Titan بالكامل — وهو ما يجعلها إطاراً لا مجرد
   wrapper، بالضبط كما تملك اليوم شكل `ctx.reply()`/`ctx.send()` فوق
   Bot API.

2. **ملكية البروتوكول المشفَّر نفسه (raw protocol ownership)** — إعادة
   تطبيق تفاوض المفاتيح (DH)، تشفير AES-IGE، تسلسل TL-schema (آلاف
   الأنواع)، إدارة الجلسة الثنائية على مستوى البِتّات، الترحيل بين
   مراكز بيانات Telegram (DC migration). هذا **ليس نفس نوع العمل** الذي
   قامت به Titan في `titan.telegram.Telegram` — ذاك عميل HTTP+JSON بسيط
   (أقرب لعشرات الحقول)، وهذا تطبيق بروتوكول تشفير حقيقي (بحجم مشروع
   Pyrogram أو Telethon كاملاً، آلاف ساعات العمل والمراجعة الأمنية).

**المخاطر الحقيقية في إعادة تطبيق الطبقة المشفَّرة من الصفر:**

- **مخاطرة أمنية فعلية، لا نظرية.** أخطاء دقيقة في تطبيق IGE mode أو
  توليد مفتاح المصادقة (auth key) ليست "أخطاء برمجية عادية" — هي أخطاء
  تكسر ضمانات التشفير بصمت (تسريب جلسة، قابلية إعادة تشغيل الرسائل
  replay، هجمات downgrade). هذه فئة أخطاء لا يكشفها test suite عادي —
  تحتاج مراجعة أمنية متخصصة لم تكن الفريق يحتاجها من قبل على جانب
  Bot API (حيث النقل عبر HTTPS القياسي فقط، بلا تشفير مخصص لـ Titan).
- **حجم صيانة غير قابل للمقارنة.** TL-schema يتغيّر مع كل تحديث لـ
  Telegram API layer (طبقات عدّية بمرور السنين)، بعشرات/مئات الأنواع.
  مقابل Bot API حيث الحقول مستقرة نسبياً وقليلة العدد. صيانة تطبيق
  MTProto كامل تعني فعلياً صيانة "عميل Telegram كامل"، لا "طبقة ترجمة".
- **مخاطرة توسّع النطاق (scope creep) الحقيقية.** "نبني فقط ما تحتاجه
  Titan" يبدو محدوداً، لكن من الناحية العملية مستخدمي Userbot يتوقعون
  عادة قدرات واسعة (وسائط، ملفات، مكالمات، جهات اتصال) تدريجياً —
  والحد الفاصل بين "طبقة ترجمة صغيرة" و"عميل Telegram كامل منافس
  لـ Pyrogram" يذوب تدريجياً بدون قرار واعٍ في كل خطوة.
- **لا يوجد مكسب معماري حقيقي من إعادة تطبيق التشفير بالذات.** ملكية
  Titan لجانب Bot API قيّمة لأنها *بسيطة، مستقرة، ومنخفضة المخاطر* —
  خط HTTP+JSON. النسخة المكافئة في MTProto ليست بسيطة ولا منخفضة
  المخاطر بطبيعتها، فالمقارنة "لو ملكنا Bot API فيجب أن نملك MTProto
  بنفس الطريقة" تتجاهل أن الطبقتين ليستا من نفس رتبة التعقيد.

**التوصية الهندسية:** الملكية المهمة فعلياً — تلك التي تحقق هدف "Titan
تمتلك أسطح Telegram التي تدعمها" — هي ملكية **السطح** (القسم 1 أعلاه)،
لا ملكية **البروتوكول المشفَّر الخام** (القسم 2). الصياغة الأدق لهذا
الفرق (صياغة مُحسَّنة بعد مراجعة نقدية، 2026-07-12):

> **Titan تملك السطح الدلالي (semantic surface)، لا سلك بروتوكول
> Telegram نفسه (the Telegram wire protocol).**

هذه الصياغة أدق من "Surface Contract مقابل Protocol Implementation"
لأنها تكشف طبقة وسطى كانت غائبة عن التحليل الأول — الملكية ليست
ثنائية (تشفير خام / سطح Titan) بل ثلاثية:

```
Protocol Implementation   ← ليست Titan بأي حال
  (AES-IGE, TL parsing, auth key exchange, DC migration)
        ↓
Protocol Client           ← منطقة رمادية — تحتاج قراراً صريحاً عند التصميم
  (session lifecycle, reconnect, entity cache, dialogs, raw requests)
        ↓
Titan Surface              ← يجب أن تملكه Titan بالكامل
  (كيف يظهر Userbot داخل Titan، ترجمة الأحداث، تعامل Context، error model)
```

**الطبقة الوسطى (Protocol Client) هي المنطقة التي لم تُحسم بعد
بوضوح.** إدارة الجلسة، إعادة الاتصال، وذاكرة التخزين المؤقت للكيانات
(entity cache) ليست "بروتوكول خام" بالمعنى الضيق، لكنها أيضاً ليست
جزءاً واضحاً من "السطح الدلالي" الذي يراه مستخدم `ctx`. هل تُترك بالكامل
للمكتبة الخارجية (Pyrogram/Telethon تديرها داخلياً)، أم تحتاج Titan
لطبقة تحكم خاصة بها فوقها (مثلاً: سياسة إعادة الاتصال الموحّدة مع بقية
Titan)؟ هذا سؤال يُحسم عند تصميم الـ ADR، لا الآن — لكنه يستحق أن
يكون بنداً صريحاً في ذلك التصميم، لا افتراضاً ضمنياً. الحد الفاصل بين
الطبقة الوسطى والسطح يجب أن يكون صريحاً ومسمّى بوضوح (تماماً كمبدأ
"حد الترجمة" في الخيار هـ):

```
[ Pyrogram/Telethon أو مكتبة تشفير/نقل منخفضة المستوى ]  ← بروتوكول خام فقط
                    ↓
[ Titan MTProto Surface — تملكه Titan بالكامل ]           ← جلسات، ترجمة، سياسات
                    ↓
[ Titan Core — Router → Middleware → Context → Handlers ] ← بلا تغيير
```

هذا **ليس** نفس فكرة "تكامل خارجي" المرفوضة أعلاه، بل تضييق الاعتماد
الخارجي إلى أضيق طبقة ممكنة (المصادقة/التشفير/النقل الخام فقط)، مع
إبقاء كل ما له معنى معماري — الجلسة، الترجمة، القرارات، التكامل مع
Core — مملوكاً لـ Titan بالكامل. كما يجب تفضيل استخدام هذه المكتبات في
**نمط "خام" (raw mode)** إن وُجد (كلتا المكتبتين تسمحان باستدعاء طبقة
النقل/التشفير دون نظام Dispatch/Plugins الخاص بهما) — لتجنّب "framework
فوق framework" (حلقتا معالجة متنافستان، كل منهما تحاول التحكم بالحدث).

**لا يغيّر هذا شيئاً في فلسفة Unified Telegram Surface (الخيار هـ):**
سؤال "من يملك التشفير الخام" مستقل تماماً عن سؤال "مسار معالجة واحد
أم مساران". الخيار (هـ) يبقى صحيحاً بغض النظر عن الجواب — لأن نقطة
الالتقاء تبقى نفسها: أي مصدر (Bot API أو MTProto Surface المملوك لـ
Titan) ينتهي عند `feed_update()` بنفس الشكل الداخلي، بلا تغيير في
`Router`/`Middleware`/`Context`.

**الخطوة العملية التالية المقترحة (لا تزال تحقيقاً/تصميماً، لا كوداً):**
قبل حسم "نبني الجلسات والترجمة بالكامل بأنفسنا"، يستحق فحص عملي قصير
(spike تصميمي على الورق، لا كود إنتاج) لتحديد: هل توجد مكتبة MTProto
منخفضة المستوى بما يكفي (تتعامل مع التشفير/النقل فقط، بلا دورة حياة
أو dispatcher خاص بها) لتُستخدم كـ "بايتات مشفّرة" مكافئة لـ `aiohttp`
في القياس؟ أم أن أضيق ما هو متاح فعلياً (في Pyrogram/Telethon) لا يزال
يحمل افتراضات بنيوية (event loop خاص، إدارة جلسة مدمجة) تتعارض مع
تكامله كـ"طبقة نقل خام" فقط تحت طبقة Titan المملوكة؟ **هذا الفحص نفسه
جزء من تصميم الـ ADR، وليس تنفيذاً — ولا يزال محكوماً بالبوابة: لا كود
Userbot قبل تصميم واضح لطبقة الترجمة (القسم 5).**

### 4. مفاهيم CONTRACT.md مبنية بالكامل على مفردات Bot API

`CONTRACT.md` يستخدم `chat_id`, `user_id`, `callback_data`, `commands
via BotFather` كمفردات أساسية بلا استثناء. لا يوجد قسم واحد يتحدث عن
"سطح اتصال" كمفهوم منفصل عن "بوت". أي دعم حقيقي لـ Userbot يعني إما:
- تعميم المفردات (breaking change ضخم في التوثيق، وربما في الأسماء
  نفسها)، أو
- الإبقاء على CONTRACT كما هو باعتباره "عقد Bot API" فقط، وإضافة
  CONTRACT منفصل (أو ملحق) لسطح Userbot — وهذا يعني أن Titan تصبح
  **مظلة لإطارين شبه منفصلين**، لا إطاراً واحداً بسطحين.

هذا الاحتمال الأخير يمس مبدأ "Depth is expressed through object
hierarchy, not package splits" الموثَّق في `design_notes.md` (مبدأ
[confirmed] رقم 2) — والذي يُصرِّح صراحة: *"If Titan ever splits into
subpackages for distinct platforms or concerns, this principle would
need to be revisited."* Userbot Support قد يكون بالضبط تلك اللحظة.

**اقتراح ثالث للنقاش (2026-07-12) — عقد طبقي بدل عقد واحد أو عقدين
منفصلين:** لا تعميم المفردات بالكامل (خيار breaking)، ولا عقد منفصل
كلياً لكل سطح (خيار "إطارين تحت مظلة") — بل تقسيم `CONTRACT.md` نفسه
إلى طبقات صريحة.

**تصحيح على شكل التقسيم بعد مراجعة نقدية (2026-07-12):** التقسيم
الأول المقترح (`Core` / `Bot API` / `MTProto` بجانب بعضها كأقسام
مستوى واحد) غير دقيق مفهومياً — لأنه يضع "Core" كأنها منصة ثالثة
موازية للسطحين، بينما **Core ليست منصة، بل هي العقد نفسه**؛ السطوح
هي التطبيقات عليه. الشكل الأصوب أربع طبقات هرمية، لا ثلاثة أقسام
متوازية:

```
CONTRACT.md
    1. Interaction Contract      (العقد العام: ما هو "تفاعل" في Titan أساساً)
    2. Surface Contract
        2.1 Bot API Surface      (خاص: callback_query, BotFather commands)
        2.2 MTProto Surface      (خاص: user session, ...)
    3. Capability Rules          (متى تكون قدرة مشتركة، ومتى تبقى خاصة بسطح)
    4. Failure Rules             (كيف يفشل استدعاء غير مدعوم على سطح معيّن)
```

بهذا الشكل Titan لا تصبح "إطارين"، بل إطار واحد **بعقد هرمي** — القسم
الأول (`Interaction Contract`) يبقى مصدر الحقيقة الوحيد لما هو عام
(بما يحافظ على مبدأ "لا انقسام إلى حزم منفصلة لكل منصة")، والأقسام
2.x توثّق ملحقات صريحة لكل سطح، بينما 3 و4 يحسمان القواعد العامة التي
تحدد تصنيف أي قدرة جديدة مستقبلاً (بدل تكرار الجدل حالة بحالة). هذا
يحل التوتر بين "لا تغيير جذري في CONTRACT" و"لا خلط مفاهيم سطحين
مختلفين تحت مفردات واحدة" في آن. لا يزال هذا **مقترحاً للنقاش**، لا
قراراً — يُحسم شكله النهائي عند تصميم الـ ADR.

### 5. الفروقات في القدرات لا يمكن إخفاؤها خلف واجهة موحّدة بأمان

بعض قدرات Bot API لا معنى مباشراً لها في Userbot (`setMyCommands` —
لا "قائمة أوامر" لحساب مستخدم عادي)، والعكس صحيح (تصفح سجل دردشة كامل،
قراءة رسائل قديمة قبل انضمام البوت — مستحيل لبوت Bot API عادي). واجهة
`ctx` موحّدة تماماً بين الاثنين ستُخفي هذه الفروقات وتُنتج **صمتاً
جديداً** يشبه بالضبط ما تم توثيقه في `silent-failures.md` — استدعاء
method غير مدعوم في backend معين يجب أن يفشل بوضوح (`TitanError`)، لا
يُتجاهل أو يُعيد `None` بصمت.

## بدائل معمارية مبدئية (بدون ترجيح — للنقاش فقط)

**أ. Titan Core تبقى Bot-API-only، وUserbot يصبح مشروعاً/حزمة منفصلة**
تستهلك `titan.playground.feed_update()` أو ما يعادلها كنقطة دخول،
لكن بدون أي كود MTProto داخل `src/titan/`.
✅ لا يمس CONTRACT الحالي إطلاقاً. لا مخاطرة قانونية/سياسية داخل
الحزمة الرسمية. ⚠️ لا "دعم Userbot" حقيقي بمعنى first-class —
مجرد نقطة تكامل.

**ب. حزمة إضافية رسمية (`titan-userbot` أو `titan.extras.userbot`)**
تُطبِّق transport MTProto (فوق Pyrogram/Telethon) وتترجم لأحداث
Titan، بنفس نمط `titan.extras` الحالي (اختياري، غير مُحمَّل افتراضياً).
✅ يحافظ على Core نظيفاً (شبيه بمبدأ AliasMap/AskManager الحاليين
في extras). ⚠️ لا يزال يحتاج مراجعة كل نقاط الالتصاق أعلاه (خاصة
CONTRACT ومفاهيم القدرات المختلفة) — العمل الحقيقي لا يختفي، فقط
يُعزَل عن Core.

**ج. تعميم Core بالكامل حول "سطح اتصال" كمفهوم صريح**
(`TelegramSurface` مجرّد، Bot API وMTProto مجرد تطبيقين له) قبل أي
تنفيذ لأي منهما.
✅ الأكثر اتساقاً معمارياً على المدى الطويل. ⚠️ الأكبر أثراً وتكلفة —
يمس `bot.py`, `ctx.py`, `telegram.py`, `adapter.py`, و`CONTRACT.md`
بالكامل قبل أن يُضاف أي backend فعلي، وقد يكون تعميماً سابقاً لأوانه
(YAGNI) إذا ظل Userbot الاستخدام الوحيد المطروح.

**د. لا شيء الآن — توثيق الموقف الحالي بصراحة في CONTRACT**
(*"Titan هي إطار عمل لبوتات Telegram Bot API. دعم Userbot، إن وُجد
مستقبلاً، سيكون عبر تكامل خارجي لا عبر Core"*) وإغلاق #10 كـ "قرار
نطاق واعٍ" بدل تركه معلقاً إلى أجل غير مسمى.
✅ يُنهي الغموض الحالي فوراً بأقل تكلفة. ⚠️ يُغلق باباً قد يكون
مهماً لنمو المشروع لاحقاً؛ قرار كهذا يستحق نقاشاً واعياً لا إغلاقاً
افتراضياً بسبب التعب من التحقيق.

**هـ. Unified Telegram Surface — مسار معالجة واحد، أكثر من مصدر اتصال**
*(أُضيف 2026-07-12 بناءً على مراجعة معمارية)*

الفكرة: لا تُقسَّم Titan إلى إطارين (كما في ب) ولا يُعاد تعميم Core
بالكامل سلفاً بلا حاجة فعلية (كما في ج). بل تبقى Titan إطاراً واحداً
بخط معالجة واحد:

```
Bot API   ─┐
            ├──► Titan Core ──► Router ──► Middleware ──► Context ──► Handlers
MTProto/UB ─┘
```

**ما يكون مشتركاً في Core:** نموذج Update، Context، Router، Middleware،
معالجة الأخطاء، دورة الحياة (lifecycle/entry points مثل `feed_update`)،
أدوات الاختبار (Playground)، والفلسفة العامة (explicit over implicit،
ctx كسطح أساسي داخل handler).

**ما يبقى صريحاً وغير مُوحَّد قسراً:** القدرات الخاصة بكل سطح تبقى
مكشوفة بأسمائها الحقيقية، لا مخفية خلف واجهة موحّدة وهمية. مثال:
`ctx.reply()` مشتركة، لكن `ctx.set_commands()` قد تبقى متاحة فقط على
Bot API، و`ctx.read_history()` (مثلاً) متاحة فقط على سطح Userbot.
استدعاء قدرة غير متاحة على السطح الحالي يجب أن يفشل بوضوح (`TitanError`)
تماشياً مع فلسفة Silent Failures — لا يُعاد `None` بصمت ولا تُخفى
القدرة الناقصة.

**الفرق عن (ب) والفرق عن (ج):**
- عن (ب): لا حزمة/إضافة منفصلة تحمل منطق ترجمة كامل بعيداً عن أعين
  Core — الترجمة تحدث عند حدود الدخول (نفس نقطة `feed_update`)، ونتيجتها
  تدخل **نفس** خط المعالجة الذي يستخدمه Bot API اليوم، لا خط موازٍ.
- عن (ج): لا تعميم شامل لكل ملف في Core دفعة واحدة قبل وجود حاجة فعلية.
  التعميم يُطبَّق فقط عند حدود الترجمة (كيف يتحول الحدث الخام إلى
  Update)، لا في فلسفة `ctx`/`Router`/`Middleware` نفسها — وهذه الأخيرة
  عامة بما يكفي فعلاً (انظر التحليل أدناه).

✅ يحافظ على "إطار واحد" (يتوافق مع مبدأ "Depth is expressed through
object hierarchy, not package splits") بدل الانقسام إلى إطارين شبه
منفصلين. ✅ لا يفرض تعميماً سابقاً لأوانه على كل Core. ⚠️ يتطلب انضباطاً
صارماً في الحفاظ على الخط الفاصل بين "المشترك" و"الخاص بالسطح" — أي
تسرّب لمفهوم Bot-API-specific إلى الطبقة "المشتركة" (أو العكس) يُفسد
الفكرة تدريجياً.

## تحليل الخيار (هـ) — إجابات تحقيقية على الأسئلة الأربعة

هذه إجابات تحقيقية (تصف الوضع الحالي وتُقدِّر الفجوة)، **لا قرارات
نهائية**. الغرض تضييق مساحة النقاش قبل أي ADR، لا استبدال الحاجة إليه.

### 1. هل Update/Context الحاليان abstraction كافٍ للسطح الموحّد؟

**جزئياً — وبتوزيع غير متساوٍ بين الطبقتين.**

- **طبقة الأفعال (`ctx.reply`, `ctx.send`, ...) قريبة من الكفاية.**
  هذه الطرق تستدعي `self._api.send_message(...)` بمعاملات عامة
  (`chat_id`, `text`, ...) على كائن `self._api` بلا أي `isinstance`
  check. `titan.playground.RecordingTelegram` يبرهن عملياً أن أي
  كائن duck-typed يطبّق نفس التوقيعات يعمل بدلاً من `Telegram` الحقيقية.
  هذه الطبقة **جاهزة تقريباً** لتقبل transport آخر دون تعديل جوهري.

- **طبقة البيانات (`titan.update.Update`) غير كافية — هذا الفجوة
  الحقيقية.** `Update.__init__` وكل الخصائص (`_user`, `_chat`,
  `get_message`) تستخدم `raw.get("message")`, `raw.get("callback_query")`,
  `raw.get("new_chat_members")` مباشرة — أي أن الصنف نفسه **يفهم** شكل
  JSON الخاص بـ Bot API، لا أنه يستقبل شكلاً موحّداً جاهزاً. هذا يعني
  أن `Update` اليوم ليست "طبقة تجريد محايدة تجاه المصدر" كما بدت من
  النظرة الأولى — هي **محلِّل (parser) خاص بـ Bot API** يُصادف أنه
  منفصل في ملف مستقل.

  لجعلها كافية لسطح موحّد، التغيير المطلوب ليس في `Context` (خصائصه
  الظاهرة: `text`, `chat_id`, `user_id`, `message_id`, `callback_data`
  تبقى كما هي)، بل في **مسؤولية `Update`**: يجب أن تتحول من "تُفسِّر
  JSON خام بنفسها" إلى "تستقبل حقولاً موحّدة جاهزة من طبقة ترجمة تسبقها"
  — أي أن كل surface (Bot API أو MTProto) يملك مترجماً خاصاً به
  (`from_bot_api_update(raw)`, `from_mtproto_update(raw)`) يُنتج نفس
  الشكل الداخلي الذي يقرأه `Update`/`Context` دون تغيير.

  **الخلاصة:** لا حاجة لتغيير الواجهة الظاهرة لـ `Context` (وهذا خبر
  جيد لتوافق Bot API الحالي)، لكن حاجة حقيقية لإعادة هيكلة *من يبني*
  `Update` — نقل معرفة "شكل Bot API" من داخل `Update` نفسها إلى طبقة
  ترجمة صريحة قبلها. هذا تغيير داخلي (internal refactor)، لا تغييراً
  في العقد الظاهر — لكنه ليس تافهاً، ويلمس كل مكان يبني `Update(raw)`
  مباشرة (`bot.py::_handle_update`، `feed_update`، اختبارات كثيرة).

- **تصحيح مهم على الصياغة أعلاه (2026-07-12):** المشكلة ليست فقط
  "`Update` تفسّر JSON خام" — بل أعمق قليلاً: **`Update` تقوم بدورين
  مختلطين في آنٍ واحد**: (1) نموذج بيانات داخلي (internal event model)
  تقرأه `Context`، و(2) محلِّل مصدر خارجي (source parser) خاص بـ Bot
  API. السؤال الأدق ليس "هل `Update` تستقبل Bot API أو MTProto؟" بل
  **"هل `Update` نفسها هي النموذج الداخلي، أم مجرد غلاف (wrapper) حوله
  خُلط بمحلِّل خارجي؟"** — وهذا خلط مسؤوليات (mixed responsibility)
  حقيقي، لا مجرد نقص في التغطية.

  المسار البديل الذي يستحق البحث عند التصميم الفعلي (لا حسم الآن):
  فصل نموذج حدث داخلي مستقل صراحةً — مثلاً وحدة `titan.events` بأنواع
  مثل `MessageEvent`/`CallbackEvent`/`ChatMemberEvent` — بحيث يكون كل
  مترجم سطح (`BotAPITranslator`, `MTProtoTranslator`) منتِجاً لهذا
  النموذج المستقل، و`Update`/`Context` يقرآنه دون معرفة أي مصدر:

  ```
  BotAPITranslator  ─┐
                      ├──► InternalEvent ──► Update/Context ──► Router
  MTProtoTranslator ─┘
  ```

  بدل الصورة الأبسط (والأخطر على المدى الطويل) التي تجعل `Update`
  نفسها نقطة التقاء المصدرين مباشرة (`BotAPI raw → Update ← MTProto raw`)
  — لأن هذه الصورة الثانية تُبقي `Update` حاملة "ندوب" تاريخها كمحلِّل
  Bot API حتى بعد الفصل الشكلي، بدل أن تكون نموذجاً نظيفاً من الأساس.
  هذا يبقى **بديلاً مطروحاً للنقاش**، وليس قراراً — الاختيار بين
  "الإبقاء على `Update` كنموذج داخلي بعد تفريغها من منطق Bot API" و
  "استخراج `InternalEvent` منفصل تماماً" هو بالضبط ما يُحسم في تصميم
  الـ ADR (انظر السؤال 1 في القائمة الختامية).

### 2. ما الحد الأدنى من MTProto لنقول إن Titan تدعمه فعلاً؟

**استقبال رسائل جديدة + إرسال رسائل نصية، مترجَمة إلى نفس شكل Update
— هذا وحده كافٍ ليُقال "Titan تدعم Userbot" بمعنى الحد الأدنى المسؤول.**

المعيار المقترح للحد الأدنى (لا أكثر ولا أقل، حتى لا يكون الوعد أكبر
من الواقع):
- استقبال حدث "رسالة جديدة" من MTProto (أي شكل كان — `UpdateNewMessage`
  أو ما يعادله في المكتبة المستخدمة) وترجمته إلى `Update` بنفس الحقول
  الأساسية (`text`, `chat_id`, `user_id`, `message_id`).
- `ctx.reply()` / `ctx.send()` يعملان فعلياً على هذا السطح (إرسال رسالة
  نصية بسيطة عبر حساب Userbot).
- Router/Middleware/error handling يعملون بلا أي تعديل — هذا **شرط
  نجاح**، لا مجرد نتيجة مرغوبة؛ إذا احتاجت هذه الطبقات أي تعديل خاص
  بـ MTProto فهذا يعني أن الفصل الذي افترضناه في الخيار (هـ) لم ينجح.

كل ما بعد ذلك (`callback_data` المكافئ، إدارة الأعضاء، `read_history`،
الوسائط) هو **توسيع تدريجي لاحق**، لا شرطاً لإعلان "الدعم موجود". هذا
يمنع سيناريو "دعم جزئي معلن كدعم كامل" الذي يُنتج صمتاً جديداً بمجرد
أن يستدعي مطوّر قدرة غير موجودة فعلياً على هذا السطح.

### 3. هل يحتاج Unified Surface إلى Titan v2، أم توسّعاً تدريجياً؟

**توسّع تدريجي — بشرط واحد صريح: أن يبقى تغيير `Update` refactor
داخلياً وغير كاسر للعقد الظاهر.**

الحجة لصالح التدريجي:
- `Context` الظاهرة، `Router`، `Middleware`، `feed_update` لا تحتاج
  تغييراً في توقيعاتها العامة حسب التحليل في السؤال (1) — التغيير
  محصور في *من يبني* `Update`، لا في شكله الظاهر.
- سابقة `RecordingTelegram` تُثبت أن Titan تدعم فعلياً استبدال transport
  دون كسر أي شيء — هذا نمط تدريجي حدث من قبل، لا نظري.
- CONTRACT الحالي (مفردات Bot API) لا يحتاج نسخاً موازياً فوراً إن بقيت
  القدرات الخاصة بكل سطح موصوفة بوضوح كإضافات، لا كتعديل على المفردات
  القائمة.

الشرط الذي يحوّله إلى v2 فعلياً: إن اكتُشف أثناء التنفيذ الفعلي أن
مفردة أساسية في CONTRACT (مثل `chat_id` أو مفهوم "الرسالة") لا تصمد
معنى موحّداً بين السطحين وتحتاج إعادة تسمية أو كسراً حقيقياً — عندها
هذا التقدير التدريجي يجب أن يُعاد فتحه، لا أن يُفرض عليه استمرار العمل
التدريجي بالقوة.

### 4. أين الحد الفاصل بين Core المشترك والقدرات الخاصة بكل سطح؟

**المعيار المقترح: القدرة تدخل Core المشترك فقط إذا كان لها معنى
متطابق فعلياً (لا مجرد اسم متشابه) على السطحين. أي قدرة معناها يختلف
جوهرياً — تبقى صريحة وخاصة بسطحها، ولا تُعمَّم قسراً لتبدو موحّدة.**

بناءً على الملفات الحالية:

| الطبقة | الحكم | السبب |
|---|---|---|
| `Update` (بعد الترجمة)، `Context` الأساسية (`text`, `chat_id`, `user_id`, `message_id`, `reply`, `send`) | **Core مشترك** | معنى متطابق فعلياً: "رسالة نصية في محادثة من مُرسل" — لا فرق جوهري بين البروتوكولين هنا. |
| `Router`, `Middleware`, معالجة الأخطاء, lifecycle (`feed_update`) | **Core مشترك — بلا تغيير أصلاً** | هذه الطبقات لا تعرف شكل JSON من الأساس (تعمل فوق `Context`/`Handler` فقط) — التحقيق الحالي لم يجد أي التصاق بها بـ Bot API. |
| `answer_callback_query` / `set_my_commands` (وشكل `callback_query` الرسمي بالتحديد) | **خاص بـ Bot API فقط (تطبيق الحالي)** | مفاهيم رسمية Bot API لا مكافئ حرفي لها في تفاعل Userbot عادي — تعميم *شكلها الحالي* قسراً يُنتج واجهة وهمية. لكن المفهوم الأعمّ خلفها — "حدث تفاعل مستخدم بزر/إجراء داخل واجهة" (interaction callback/event) — قد لا يكون خاصاً بالضرورة إذا ظهرت مستقبلاً أسطح أخرى (مثل UI plugin) تحتاج مفهوماً مشابهاً؛ الاسم والشكل الحاليان لا يُربطان بالمفهوم النهائي مسبقاً. |
| قراءة سجل كامل قبل انضمام الحساب، إرسال من "حساب شخصي" لا "بوت" | **خاص بـ Userbot فقط** | لا مكافئ في Bot API — نفس المنطق بالعكس. |
| `bot.telegram.*` (`TelegramAdapter`) | **يحتاج قراراً صريحاً لاحقاً** | حالياً موصوف كـ "وصول كامل لـ Bot API خارج ctx" — إن وُجد MTProto surface فعلياً، هل يحصل على مكافئ (`bot.userbot.*`) بنفس الفلسفة، أم يبقى `bot.telegram` خاصاً بذلك السطح فقط ويُضاف اسم مختلف للآخر؟ هذا سؤال تسمية/تنظيم يستحق حسماً عند التنفيذ، لا الآن. |

**الخلاصة العملية:** الخط الفاصل ليس "كل شيء في `ctx` مشترك، كل شيء
في `bot.telegram` خاص" — الخط الفاصل الحقيقي هو **تطابق المعنى**، لا
موقع الكود. بعض ما في `ctx` اليوم (`callback_data`) هو فعلياً خاص
بـ Bot API ويحتاج نقلاً منطقياً (لا جسدياً بالضرورة، توثيقاً على الأقل
باعتباره Bot-API-only) إن تحقق الخيار (هـ) فعلياً.

## الاتجاه المعماري المفضَّل — Pending ADR (2026-07-12)

**الحالة: اتجاه مبدئي مختار، وليس قراراً معمارياً نهائياً.** لا يوجد
ADR بعد، ولا كود. ما يلي هو الاتجاه الذي سيُبنى عليه الـ ADR لاحقاً،
بعد حسم تصميم طبقة الترجمة (البند 5 أدناه) — وليس بديلاً عنه.

الاتجاه المفضَّل حالياً: **الخيار (هـ) — Unified Telegram Surface.**

لا فصل إلى إطارين، لا حزمة/مشروع منفصل لـ Userbot، لا تعميم شامل لكل
Core قبل الحاجة. Titan تبقى إطاراً واحداً بمسار معالجة واحد:

```
Bot API  ─┐
           ├──► Titan Core ──► Router ──► Middleware ──► Context ──► Handlers
MTProto  ─┘
```

هذا **اتجاه مفضَّل (Preferred Direction)**، لا Architecture Decision
نهائي. الـ ADR يُكتب لاحقاً، عندما يكون تصميم طبقة الترجمة (البند 3
أدناه) محدَّداً بدقة كافية ليصبح التزاماً قابلاً للمراجعة والنقض —
وحتى ذلك الحين يبقى هذا الاتجاه قابلاً للتعديل أو النقض إذا كشف
تصميم الترجمة الفعلي عكس ذلك.

**القاعدة الحاكمة لكل ما يلي:**
> كل ما له معنى متطابق فعلياً بين Bot API وMTProto → Core مشترك.
> كل ما هو خاص بسطح واحد → يبقى صريحاً، مسمى بوضوح، ولا تُخترع له
> abstraction وهمية لمجرد توحيد الشكل.

**سؤال مفقود من النسخة الأولى — أهم سؤال في هذا التحقيق (2026-07-12):**
كل ما سبق افترض أن `Context` تمثل "تفاعلاً مع Telegram". لكن هناك سؤال
أعمق لم يُطرح صراحة: **هل `Context` تمثّل "Telegram" أم "تفاعلاً"
(interaction) بشكل عام؟**

الخصائص الحالية (`ctx.chat_id`, `ctx.user_id`, `ctx.message`) تبدو
خاصة بـ Telegram بحكم أسمائها. لكن معناها الفعلي أعمّ: *"طرف يبدأ
تفاعلاً، طرف يستقبله، رسالة/فعل بينهما"* — وهذا أقرب لمفردات مثل
`ctx.actor`, `ctx.target`, `ctx.interaction`. الفرق ليس تجميلياً: Bot
API تتكلم عن *"مستخدم يرسل رسالة إلى بوت"*، وMTProto تتكلم عن *"حساب
يستقبل رسالة"* — المعنى قريب جداً، لكن **الهوية اللغوية مختلفة**، وهذا
يظهر بوضوح فقط عند وجود سطح ثانٍ حقيقي.

**لا يعني هذا تغيير أسماء `ctx` الآن** — `ctx.chat_id`/`ctx.user_id`
تبقى كما هي، والتوافق مع Bot API الحالي لا يُمس. لكن الـ ADR المستقبلي
يجب أن يجيب صريحاً عن هذا السؤال الهوياتي: هل Titan تريد أن تكون
**Telegram Bot Framework** (تحسم مفرداتها حول "بوت" و"Bot API")، أم
**Telegram Interaction Framework** (مفرداتها حول "تفاعل" بغض النظر عن
كونه بوتاً أو حساباً)؟ هذا هو نفس السؤال المطروح في اقتراح إعادة صياغة
عنوان الـ ADR (انظر البند 6 أدناه) — لكن على مستوى مفردات `Context`
نفسها، لا فقط عنوان الوثيقة.

### 1. Core المشترك — القائمة (بحكم مراجعة الكود الفعلي، لا افتراضاً)

| العنصر | لماذا يدخل Core |
|---|---|
| **Update بعد الترجمة** — الحقول المعيارية الداخلية: `text`, `chat_id`, `user_id`, `username`, `message_id`, `reply_to_message_id`, `new_members`, `left_member` | معنى متطابق فعلياً: "رسالة نصية من مُرسل داخل محادثة" لا يختلف جوهرياً بين البروتوكولين |
| **Context الأساسية** — `ctx.text`, `ctx.chat_id`, `ctx.user_id`, `ctx.message_id`, `ctx.sender`, `ctx.chat`, `ctx.message`, `ctx.reply()`, `ctx.send()` | تعمل فعلاً فوق `Update` الموحّدة + transport مُستبدَل بـ duck typing — لا حاجة لتغييرها |
| **Router** (`on`, `command`, `callback` registration) | لا معرفة بشكل JSON من الأساس — عمل تنظيمي بحت، تحقّقتُ أنه لا يلمس raw dict إطلاقاً |
| **MiddlewareChain / Middleware** | تعمل فوق `Context` فقط، لا فوق شكل الحدث الخام |
| **معالجة الأخطاء** (`_handle_error`, `TitanError`) | عامة بالفعل، لا التصاق بمصدر الحدث |
| **نقطة الدخول** (`feed_update()`) | هذا هو "الدرزة" الرسمية بين أي مصدر والمعالجة الداخلية — موجودة أصلاً منذ ADR-011 |
| **أدوات الاختبار** (Playground / نمط `RecordingTelegram`) | أثبتت عملياً أن استبدال transport لا يكسر شيئاً |
| **الفلسفة ومفردات CONTRACT للأجزاء المشتركة** | لا تحتاج تغييراً |

### 2. خاص بكل Surface — يبقى صريحاً ولا يُوحَّد

**خاص بـ Bot API (مؤكد من الكود الحالي، كتطبيق حالي لا كمفهوم نهائي):**
`ctx.callback_data`, `ctx.callback_id` (مبنيان كلياً على شكل
`callback_query` الرسمي الحالي)، و`TelegramAdapter.set_my_commands` /
`delete_my_commands` / `get_my_commands` / `answer_callback_query` —
هذه التطبيقات بالذات لا مكافئ حقيقي لها في تفاعل Userbot عادي. لكن
المفهوم الأعمّ وراء `callback_data` — "حدث تفاعل مستخدم عبر إجراء/زر
داخل واجهة" — قد يتحول لاحقاً إلى مفهوم مشترك إن ظهر سطح آخر يحتاجه؛
لا نربط الاسم/الشكل الحاليين بهذا المفهوم النهائي مسبقاً.

**خاص بـ Userbot/MTProto (تقديري — لا يوجد كود بعد ليؤكده):**
قراءة سجل دردشة كامل قبل انضمام الحساب، الانضمام والتفاعل كـ"حساب
شخصي" لا "بوت"، الوصول إلى جهات الاتصال. هذه قدرات **مفترضة بناءً على
طبيعة MTProto**، ويجب التحقق منها فعلياً عند التصميم، لا افتراضها
نهائية من الآن.

**تحتاج قراراً عند التصميم الفعلي (غير محسومة الآن):**
- هل `is_channel_post` / رسائل القنوات مفهوم مشترك فعلاً (تيليجرام
  تدعم القنوات على السطحين) أم أن دلالتها تختلف كفاية لتُصنَّف خاصة؟
  يحتاج فحصاً وقت بناء أول مترجم MTProto حقيقي — لا يُحسم نظرياً.
- تسمية namespace القدرات الكاملة لكل سطح: هل `bot.telegram` يبقى
  خاصاً بـ Bot API ويُضاف `bot.userbot` بجانبه، أم اسم عام أوسع؟ سؤال
  تسمية/تنظيم يُحسم عند التنفيذ، ليس الآن.
- **سؤال ناقص أضيف الآن (2026-07-12): أين تعيش القدرات الخاصة بسطح
  واحد على مستوى `ctx` نفسها، لا فقط على مستوى `bot.*`؟** الاتجاه
  الحالي يحسم "ما هو مشترك يدخل Core، الخاص يبقى خاصاً" لكن لا يحسم
  *مكان* الخاص نحوياً على `Context`. مثال ملموس: هل قدرة خاصة بسطح
  (كقراءة السجل الكامل في MTProto مستقبلاً) تكون `ctx.read_history()`
  مباشرة، أم `ctx.userbot.read_history()`، أم `ctx.surface.read_history()`؟
  المشكلة في الخيار الأول (`ctx.read_history()` مباشرة على `Context`
  المشتركة) أنه يخلق غموضاً مستقبلياً بمجرد ظهور تدرّج في الدعم: اليوم
  الوضع ثنائي (يدعمها Userbot / لا يدعمها Bot)، لكن مستقبلاً قد يظهر
  سطح ثالث (Business Bot API مثلاً) يدعم *جزءاً* من نفس القدرة —
  فتصبح method واحدة على `Context` المشتركة غير كافية للتعبير عن هذا
  التدرّج بوضوح. تجميع القدرات الخاصة بسطح تحت namespace صريح
  (`ctx.userbot.*` أو `ctx.surface.*`) أوضح لهذا السبب تحديداً — لكن
  هذا **اختيار يُحسم عند تصميم الـ ADR**، لا الآن.

  **تفضيل مُضاف بعد مراجعة نقدية (2026-07-12): `ctx.surface.*` أفضل
  من `ctx.userbot.*` — والسبب أعمق من مجرد التسمية.** "Userbot" اسم
  لحالة استخدام (استخدام حساب شخصي)، لا مفهوماً معمارياً. المفهوم
  المعماري الصحيح هو **السطح** نفسه — وتسمية namespace القدرات
  الخاصة باسم حالة استخدام (`userbot`) بدل اسم المفهوم (`surface`)
  يُثبِّت افتراضاً ضيقاً: أن "الخاص" = "خاص بـ MTProto حصراً". لكن
  `ctx.surface.*` يبقى صحيحاً حتى لو ظهر سطح ثالث لاحقاً (مثلاً
  `ctx.surface.history()`, `ctx.surface.contacts()`,
  `ctx.surface.permissions()`) دون الحاجة لاسم جديد لكل سطح جديد —
  المفهوم "قدرة خاصة بالسطح الحالي" يبقى ثابتاً، فقط محتوى `surface`
  يختلف بحسب أي backend فعلياً متصل.

### 3. أقل refactor مطلوب — بدون كسر CONTRACT الحالي

**المبدأ:** لا شيء في الواجهة الظاهرة لـ `Context`/CONTRACT.md يتغيّر.
التغيير محصور بالكامل في *من يملك مسؤولية فهم شكل الحدث الخام*.

**ملاحظة على الخطوات التالية:** الوصف أدناه يفترض — كأبسط سيناريو
عملي — أن ناتج مترجم أي سطح يبقى نفس شكل `Update` الحالي. هذا افتراض
عمل (working assumption) لتوضيح حجم التغيير الأدنى، وليس حسماً لسؤال
"من يملك النموذج الداخلي؟". البدائل الممكنة (`Translator → Update`
مباشرة، أو `Translator → InternalEvent → Update`، أو حتى
`Translator → Context Builder`) تبقى كلها مفتوحة، وتُحسم فعلياً عند
تصميم الـ ADR (انظر البند 5).

اليوم، `titan.update.Update.__init__` وخصائصها الداخلية (`_user`,
`_chat`, `get_message`) **هي نفسها** منطق "ترجمة Bot API" — لكنه مكتوب
كجزء غير منفصل من الصنف بدل أن يكون حدوداً صريحة. أقل تغيير كافٍ:

1. **عزل منطق "قراءة JSON الخاص بـ Bot API" في حدود صريحة** (مفهومياً:
   "مترجم سطح Bot API") — بدون تغيير أي حقل ظاهر لـ `Update` أو
   `Context`. بما أن هذا المنطق موجود بالفعل ومكانه فقط يتغيّر، هذا
   refactor داخلي منخفض المخاطر، ويمكن إجراؤه **الآن، قبل وجود
   MTProto فعلياً**، كخطوة تحضيرية بحتة — يثبت الاختبار الحالي (831
   اختباراً) أنه لم يكسر شيئاً.
2. **`feed_update()` تبقى نقطة الدخول الوحيدة** بغض النظر عن السطح؛
   مترجم كل سطح يقف مباشرة قبلها (مترجم Bot API اليوم، مترجم MTProto
   عند بنائه لاحقاً).
3. **لا تغيير في `Router`/`Middleware`/الشكل الظاهر لـ `Context`** —
   هذا refactor لا يلمسها إطلاقاً، وهذا هو الدليل العملي على أن الفصل
   القائم عليه الخيار (هـ) صحيح، لا افتراض نظري.

هذا يعني: **الحد الأدنى المطلوب فعلياً قبل أي كود MTProto هو خطوة
تحضيرية صغيرة ومعزولة في `Update` وحدها — لا في بقية Core.**

### 4. لا تعميم قبل الحاجة — الحدود فقط عند نقاط الترجمة

تأكيد صريح: التجريد الوحيد المُدخَل هو عند **حد الترجمة** (حدث خام ←
حقول Update المعيارية). حد النقل (`_api`) موجود أصلاً عبر duck typing
ولا يحتاج تغييراً. لا داعٍ لإدخال abstract base classes أو interfaces
جديدة في أي مكان آخر من Core حتى يوجد سطح ثانٍ حقيقي يكشف حالات فعلية
لا تكفيها هذه الحدود — تجنّباً لتعميم افتراضي (speculative generalization)
لا يخدم حاجة قائمة.

### 5. لا كود Userbot قبل تصميم واضح لطبقة الترجمة (translation layer)

قبل أي كود MTProto — ولو تجريبياً — يجب أن يُحسم أولاً (كتحقيق/تصميم،
ثم ADR، بهذا الترتيب):

- **الشكل المعياري الداخلي الدقيق** الذي يجب أن تُنتجه أي طبقة ترجمة:
  الحقول بالضبط وأنواعها (نفس حقول `Update` الحالية، أم بنية بيانات
  صريحة مستقلة — مثل `dataclass` وسيط — يقبلها `Update` بدل `raw dict`؟).
- **من يملك هذا الشكل** بعد الفصل: هل يبقى ضمنياً داخل `Update` كما
  اليوم (بعد نقل منطق Bot API فقط)، أم يُستخرج كنوع بيانات مستقل قابل
  للاختبار بمعزل عن أي سطح؟
- **التوقيع الدقيق** الذي يجب أن يلتزم به أي "مترجم سطح" مستقبلي —
  هذا هو ما سيُكتب فعلياً كـ ADR عندما يحين وقته، لا قبل ذلك.

**الخلاصة:** الاتجاه مفضَّل، لا معتمد بشكل نهائي بعد. التصميم
التفصيلي لطبقة الترجمة، ثم الـ ADR، ثم الكود — بهذا الترتيب فقط.

### 5ب. هل هذا يستدعي Titan v2؟ — لا (2026-07-12)

سؤال صريح يستحق جواباً صريحاً بدل تركه مفتوحاً: **لا، ليس بسبب Userbot
تحديداً.** استخراج نموذج حدث داخلي، إضافة مترجم لكل سطح، وإضافة
namespace للقدرات الخاصة (`ctx.surface.*`) — هذه كلها **تطور معماري
طبيعي (natural architectural evolution)**، لا كسراً للعقد العام. لا
شيء في هذا العمل يُغيّر معنى `ctx.reply()` أو يكسر توافق Bot API
الحالي. **معيار Titan v2 يجب أن يكون كسر العقد العام نفسه، لا مجرد
تنظيف مسؤوليات داخلية (internal responsibility cleanup)** — وهذا
التمييز يستحق أن يبقى صريحاً في أي مناقشة مستقبلية حول حجم هذا العمل.

### 5ج. الصياغة المرشَّحة لجوهر الـ ADR المستقبلي (مرجع فقط، ليست ADR)

هذا **ليس** الـ ADR — لا يزال يُكتب فقط بعد Phase 0.5/1/2. لكن، كمرجع
لما قد يكون جوهره لاحقاً بناءً على كل ما سبق في هذا التحقيق:

> Titan will support multiple Telegram interaction surfaces through
> explicit translation boundaries. Core processing remains
> surface-agnostic. Surface-specific capabilities remain explicit and
> are never artificially unified.

مع قرارات تابعة متوقعة (تُحسم فعلياً عند الكتابة، لا الآن):
1. فصل الـ parsing عن النموذج الداخلي (Canonical Interaction Model).
2. وجود مترجم (Translator) مستقل لكل سطح.
3. Core لا يعرف تفاصيل Bot API ولا MTProto إطلاقاً.
4. كل سطح يملك (owns) قدراته الخاصة، مسمّاة صريحاً تحت `ctx.surface.*`
   أو ما يعادله.
5. تنفيذ البروتوكول الخام (Protocol Implementation) ليس مسؤولية
   Titan — Titan تملك السطح الدلالي فقط، لا سلك البروتوكول.

### 6. عنوان الـ ADR المستقبلي — لا يجب أن يبدأ بـ "Userbot" (2026-07-12)

ملاحظة توجيهية لوقت كتابة الـ ADR فعلياً: عنوانه **لا ينبغي** أن يكون
شيئاً مثل *"Add Userbot Support"* — لأن هذا يحصر القرار في ميزة واحدة،
بينما القرار الحقيقي أوسع من ذلك. الصياغة الأدق تعكس السؤال الفعلي:
هل Titan إطار عمل لبوتات Bot API (Telegram Bot Framework)، أم إطار
لمعالجة تفاعلات Telegram بغض النظر عن سطحها (Telegram Interaction
Framework)؟ Userbot Support هو أول اختبار عملي لهذا التعريف، لا محور
القرار نفسه. عناوين مقترحة أقرب لجوهر القرار: *"Introduce Telegram
Interaction Surfaces"* أو *"Separate Telegram Event Translation From
Core Processing"*. هذا توجيه للصياغة وقت الكتابة، وليس قراراً يُتَّخذ
الآن.

### 7. الترتيب الموصى به قبل أي ADR (2026-07-12، مُحدَّث بعد مراجعة نقدية)

بدل القفز مباشرة من "اتجاه مفضَّل" إلى اختيار مكتبة MTProto، الترتيب
الأنضج المقترح:

```
Phase 0   — تحقيق معماري (هذا الملف، منتهٍ)
    ↓
Phase 0.5 — Capability Matrix: أي قدرة مدعومة على أي سطح فعلياً (جدول)
    ↓
Phase 1   — تعريف النموذج الداخلي (Internal/Canonical Event Model) ككيان محتمل مستقل عن Update
    ↓
Phase 2   — فصل مترجم Bot API عن Update (البند 3 أعلاه، أقل refactor)
    ↓
Phase 3   — كتابة الـ ADR الفعلي (Unified Telegram Surface + حدود الملكية)
    ↓
Phase 4   — Spike صغير جداً لـ MTProto (استقبال رسالة + إرسال رسالة فقط)
    ↓
Phase 5   — قرار التوسّع أو التوقف، بناءً على نتائج الـ Spike
```

**Phase 0.5 مُضافة بعد مراجعة نقدية — قبل أي تصميم، لا بعده.** جدول
صريح يقارن كل قدرة معروفة عبر السطوح المطروحة (Bot API، MTProto)
وحكمها في Core، مثلاً:

| القدرة | Bot API | MTProto | الحكم في Core |
|---|---|---|---|
| إرسال رسالة | نعم | نعم | مشترك |
| الرد على رسالة | نعم | نعم | مشترك |
| أوامر (`/command`) | نعم | لا | خاص بـ Bot API |
| قراءة سجل كامل | محدود | نعم | خاص بـ MTProto |
| callback / تفاعل زر | نعم (شكل رسمي) | مختلف الشكل | يحتاج قراراً — أقرب لمفهوم مشترك بشكلين مختلفين لا قدرة موحّدة فوراً |

**الفائدة المباشرة:** هذا الجدول يمنع تكرار خطأ "توحيد أشياء ليست
متطابقة المعنى" *قبل* أن يُكتب أي كود ترجمة أو أي ADR، لا بعد اكتشافه
أثناء البناء. هذا الجدول تمهيدي هنا ولا يُعتبر نهائياً — يُبنى بدقة
أكبر فعلياً في Phase 0.5 عند التصميم، وليس التزاماً كاملاً الآن.

**لماذا الترتيب بهذا الشكل:** الخطر الحقيقي في القفز مباشرة لاختيار
مكتبة قبل Phase 0.5/1/2 هو اكتشاف لاحق (أثناء أو بعد بناء Spike
MTProto) أن `Context` أو `Update` كانا مرتبطين ضمنياً بافتراضات خاصة
بـ Bot API لم تُكتشف في التحقيق النظري — ما يعني كتابة ADR ثانٍ لتصحيح
الأول. كل من Phase 0.5 وPhase 1 وPhase 2 عمل تحضيري منخفض المخاطر (لا
يمس CONTRACT الظاهر، مثبَت باختبارات موجودة) يقلل هذا الخطر *قبل*
الالتزام بأي مكتبة خارجية أو بأي ADR رسمي.

## أسئلة متبقية قبل كتابة الـ ADR

الخيار (هـ) هو الاتجاه المفضَّل حالياً، وليس قراراً نهائياً بعد.
الأسئلة أدناه هي ما تبقى فعلياً قبل أن يصبح هذا التصميم قابلاً لأن
يُكتب كـ ADR ثم يُنفَّذ:

1. **الشكل الدقيق لبنية الترجمة الوسيطة** (البند 5 أعلاه) — هل تبقى
   `Update` كما هي بعد نقل منطق Bot API داخلها فقط، أم تُستخرج بنية
   بيانات وسيطة صريحة (`dataclass`) يبنيها كل مترجم سطح ثم يُمرَّرها
   لـ `Update`؟ هذا يُحسم أثناء كتابة الـ ADR، لا قبله.
2. **[سؤال نطاق ملكية، أسبق من اختيار أي مكتبة]** هل MTProto surface —
   إن بُني — سطح تملكه Titan رسمياً (تطبيق البروتوكول نفسه ضمن حدود
   المشروع)، أم تكامل خارجي فوق مكتبة جاهزة (Pyrogram/Telethon)؟ هذا
   ليس سؤال "أي مكتبة نستخدم" بل سؤال فلسفي عن حدود استقلالية Titan:
   هل الاستقلالية المعلنة تشمل بروتوكولات الاتصال نفسها، أم فقط بنية
   Core (`Router`/`Context`/معالجة الأحداث)؟ الجواب هنا يُحدِّد إن كان
   Userbot Support مشروعاً بحجم طبقة ترجمة، أو مشروعاً بحجم إطار مستقل
   قائم بذاته. **يجب حسم هذا كقرار معماري مستقل — في نفس الـ ADR أو
   قبله — قبل اختيار أي مكتبة MTProto فعلياً.**
3. هل المخاطرة القانونية/السياسية المرتبطة بـ Userbot (مخالفة ToS في
   استخدامات شائعة) تخص Titan كمشروع، أم هي مسؤولية المطور الذي
   يستخدمها بالكامل — وهل هذا يحتاج إخلاء مسؤولية صريحاً في التوثيق
   بغض النظر عن القرار التقني؟
4. القرارات "غير المحسومة عند التصميم الفعلي" المذكورة في البند 2
   أعلاه (تصنيف `is_channel_post`، تسمية `bot.telegram` مقابل
   `bot.userbot`) — تحتاج فحصاً فعلياً وقت بناء أول مترجم MTProto، لا
   حسماً نظرياً الآن.

5. **[أُضيف 2026-07-12]** هل النموذج الداخلي (`titan.update.Update`)
   يبقى هو نفسه بعد تفريغه من منطق Bot API، أم يُستخرج نموذج حدث داخلي
   منفصل تماماً (`titan.events` بأنواع مثل `MessageEvent`/
   `CallbackEvent`) بحيث لا يحمل أي "ندوب" من كونه محلِّل Bot API
   سابقاً؟ (انظر الملاحظة في قسم "هل Update/Context الحاليان abstraction
   كافٍ؟" أعلاه.)
6. **[أُضيف 2026-07-12]** أين تعيش القدرات الخاصة بسطح واحد نحوياً على
   `Context` نفسها — مباشرة (`ctx.read_history()`)، أم تحت namespace
   صريح (`ctx.userbot.*` أو `ctx.surface.*`)؟ هذا مستقل عن سؤال تسمية
   `bot.telegram`/`bot.userbot` (البند 4 أعلاه)، ويحتاج حسماً منفصلاً
   بسبب احتمال ظهور تدرّج دعم جزئي بين أكثر من سطحين مستقبلاً.
7. **[أُضيف 2026-07-12]** هل يُقسَّم `CONTRACT.md` إلى عقد طبقي (Core
   Contract + Bot API Surface Contract + MTProto Surface Contract)
   بدل عقد واحد مُعمَّم أو عقدين منفصلين كلياً؟ (انظر قسم "مفاهيم
   CONTRACT.md" أعلاه.)

**لا تنفيذ، لا ADR، حتى يُحسم البند 3 من القرار المبدئي (تصميم طبقة
الترجمة)، بالإضافة إلى البنود 5–7 أعلاه، بما يكفي ليصبح مادة ADR
قابلة للمراجعة.**

## الحكم النهائي على التحقيق (تقييم خارجي، 2026-07-12)

**نقاط القوة المؤكَّدة:**
- رفض فكرة "مجرد إضافة adapter" مبكراً، والوصول للسؤال المعماري الأعمق.
- اكتشاف أن المشكلة الحقيقية في **نموذج الأحداث** (`Update`)، لا في
  طبقة النقل فقط — وهذا لم يكن واضحاً من القراءة الأولى للكود.
- الحفاظ الصريح على Core كهدف لا يُمس دون دليل عملي (لا تعميم افتراضي).
- رفض abstraction وهمية توحّد قدرات مختلفة المعنى فعلياً بين السطحين.
- الفصل الواضح بين "الملكية" و"التنفيذ" في نقاش MTProto.

**ما أُضيف في هذه الجولة قبل الانتقال لـ ADR:**
1. تمييز النموذج الداخلي (Internal Event Model) عن `Update` كغلاف —
   وطرح `titan.events` كبديل مطروح للنقاش، لا حسم.
2. تحديد أن مكان القدرات الخاصة بسطح يحتاج قراراً نحوياً صريحاً على
   `Context` نفسها (namespace)، لا فقط تصنيفها "خاصة".
3. اقتراح عقد طبقي لـ `CONTRACT.md` بدل عقد واحد أو عقدين منفصلين.
4. حسم صياغة "الملكية" كـ Surface Contract لا Protocol Implementation.
5. توجيه بعدم حصر عنوان الـ ADR المستقبلي في "Userbot" — القرار
   الحقيقي هو تعريف Titan كإطار تفاعل، لا مجرد ميزة إضافية.
6. ترتيب مراحل موصى به قبل أي ADR (تعريف النموذج الداخلي → فصل مترجم
   Bot API → كتابة الـ ADR → Spike صغير → قرار التوسّع).

لا يزال هذا الملف بالكامل توثيقاً/تحقيقاً. لا كود، لا ADR، حتى الآن.
