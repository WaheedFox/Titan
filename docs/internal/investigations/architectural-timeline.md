## الحالة الراهنة

### ما يوجد بالفعل

**`docs/decisions/README.md` — proto-timeline:**
```
| # | Title | Status |
|---|---|---|
| 001 | Keyboard Builder | Rejected |
| 002 | Actions | Accepted |
...
```
يوفر: رقم + عنوان + حالة. لا تاريخ، لا ملخص، لا Rule، لا وصول برمجي.

**ADRs — بنية متسقة:**
كل ADR يحتوي: `Status`, `Proposal`, `Investigation`, `Decision`, `Rule`, `Alternatives` (اختياري), `Consequences` (اختياري).

الـ **Rule** هو أكثر حقل دائم وقيمة: مبدأ مشتق من القرار، يُطبَّق على القرارات المستقبلية.

**مثال (ADR-004 — Error Contracts):**
```
Rule: TitanError for developer mistakes.
      TelegramError for API failures.
      Never silence either.
```

**مثال (ADR-009 — Runtime Contract Validator):**
```
Rule: If code will never work, don't allow it to be registered.
      Fail at decorator time, not at runtime.
```

### ما هو غائب

| الحاجة | الحالة |
|---|---|
| تاريخ القرار | غائب من أغلب ADRs — غير موحد |
| ملخص قصير (جملة واحدة) | غائب — المطور يقرأ الملف كاملاً |
| الـ Rule مُستخلَص | مدفون في الملف، لا يُسترجع برمجياً |
| وصول برمجي (`entry("009")`) | غائب |
| تصفية حسب الحالة | غائب |
| قراءة من Architect AI | غائب |

## تحليل النموذج الموجود: Migration Knowledge API

`titan.migration` هو النموذج المُثبَّت لـ Knowledge APIs في Titan:

```python
# ثلاث دوال فقط
frameworks()      # list[str]
concepts(fw)      # list[str]
compare(fw, c)    # ConceptMapping (frozen dataclass)
```

**الخصائص:**
- قراءة فقط — لا state، لا side effects
- frozen dataclasses — آمنة للمشاركة والكاش
- مستقلة عن runtime الـ bot — تعمل بدون token
- موثقة ذاتياً — كل field له docstring واضح
- مصممة للأدوات الذكية بالدرجة الأولى

`titan.timeline` يتبع نفس النموذج — بنية مختلفة، نفس الفلسفة.

## الحقول المتاحة في كل ADR

من فحص 9 ADRs:

| الحقل | متوفر في كل ADRs؟ | ملاحظة |
|---|---|---|
| رقم (number) | ✓ دائماً | من اسم الملف |
| عنوان (title) | ✓ دائماً | من الـ heading |
| حالة (status) | ✓ دائماً | Accepted / Rejected / Deferred |
| Rule | ✓ في أغلبها | أقوى حقل معرفياً |
| تاريخ (date) | ✗ غير موحد | ADR-009 يحتويه، الباقي لا |
| ملخص (summary) | ✗ غائب | غير موجود كحقل مستقل |
| وسوم (tags) | ✗ غائب | لا تصنيف موضوعي |

**المشكلة الجوهرية:** التاريخ غير موحد في ADRs. خياران:
- **(أ)** نضيف `**التاريخ:**` كحقل رسمي في كل ADR ونُرجعه بالـ API
- **(ب)** نعتمد الترقيم (001-009) كبديل ترتيبي — لا تاريخ حرفي

## حدود الميزة

**ما هي:**
- Knowledge API قابلة للقراءة برمجياً تُمثِّل كل ADR كـ dataclass
- بنية بيانات في `_data.py` تحتوي البيانات المُكثَّفة لكل ADR

**ما ليست:**
- لا تقرأ ملفات Markdown تلقائياً (ADRs لا تُحلَّل وقت runtime)
- لا تُنشئ توثيقاً أو HTML
- لا تُدار قاعدة بيانات
- لا تُحدَّث تلقائياً — كل ADR جديد يُضاف يدوياً في `_data.py`

## الأسئلة المفتوحة للمناقشة المعمارية

### ١ — الحقول: ما الذي نُسرِّفه في الـ API؟

**الحد الأدنى:** `number`, `title`, `status`, `rule`

**موسّع:** + `date`, `summary`, `tags`

الـ Rule هو الحقل الأعلى قيمة لـ Architect AI — مبدأ قابل للتطبيق على قرارات مستقبلية.  
`summary` يتيح قراءة سريعة دون فتح الملف.  
`tags` يتيح تصفية موضوعية (`"registration"`, `"contract"`, `"surface"`, ...).

### ٢ — التاريخ: إلزامي أم اختياري أم لا يوجد؟

- **إلزامي:** نضيف `date` لكل ADR في `_data.py` ونُرجعه. يتطلب التزاماً بتوحيد التاريخ في الملفات أو قبول أن البيانات مخزنة في _data.py فقط.
- **اختياري (`date | None`):** نُضيف للـ ADRs التي لها تاريخ، وندع الباقي `None`.
- **غائب:** نستخدم الترقيم كبديل ترتيبي — أبسط، لكن يفقد السياق الزمني.

### ٣ — اسم الوحدة: أين تسكن؟

- `titan.timeline` — الأوضح لنية الميزة
- `titan.decisions` — أكثر دقة تقنياً
- `titan.adr` — تقني لكن يحتاج شرحاً

### ٤ — دوال الـ API: ما الدوال الكافية؟

**المقترح الأساسي** (يوازي migration API):
```python
entries()              # list[ArchiveEntry] — كل القرارات
entry("009")           # ArchiveEntry — قرار محدد برقمه
by_status("accepted")  # list[ArchiveEntry] — تصفية بالحالة
```

**إضافات محتملة:**
```python
rules()                # list[str] — كل المبادئ المستخلَصة
latest(n=5)            # list[ArchiveEntry] — آخر N قرارات
```

هل `rules()` تبدو ذات قيمة كافية لتكون دالة مستقلة؟

### ٥ — `_data.py`: ملف واحد أم مُقسَّم؟

Migration API: ملف واحد (~369 سطراً).  
Timeline مع 9 ADRs وكل entry يحتوي ~5 حقول: ملف واحد كافٍ.  
مع توسع مستقبلي لـ 20+ ADR قد يحتاج تقسيماً.

### ٦ — ماذا نفعل بـ `docs/decisions/README.md`؟

هل يُحدَّث تلقائياً من `_data.py`؟ أم يبقى ملفاً مُنتَجاً يدوياً بالتزامن؟  
الأبسط: يبقى README مكتوباً يدوياً ويُحدَّث مع كل ADR جديد — كما هو الآن.
