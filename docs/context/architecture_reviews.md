مجالات المراجعة المعمارية المقترحة  
1. Error Handling & Propagation  
المشكلة الحقيقية: في PTB وaiogram، من أكثر الـ issues تكراراً على GitHub: البوت "يتجمد" دون أي رسالة خطأ. السبب الجذري: asyncio.create_task يبتلع الاستثناءات إذا لم يُلتقط الـ task. الإطارات حلّت هذا بطرق مختلفة — aiogram أضاف error middleware chain، PTB أضاف application.run_polling(error_handler=...).  
  
وضع Titan: يوجد global error_handler. لكن _chat_worker ينشئ tasks عبر asyncio.create_task — مسار الاستثناء من هذه الـ tasks يستحق تدقيقاً. السؤال ليس "هل يوجد error handler؟" بل "هل كل استثناء يصل إليه بضمان؟"  
  
2. Update Routing & Filtering  
المشكلة الحقيقية: عندما تنمو البوتات، المطورون يحتاجون routing أكثر من "أمر + نوع حدث": فلتر نوع المحتوى، فلتر المستخدم، فلتر حجم المجموعة. PTB طوّر نظام Filter قابل للتركيب (&, |, ~). aiogram بنى طبقة Filters كاملة. pyTelegramBotAPI أضاف content_types. كل إطار وصل إليه من ضغط استخدام حقيقي.  
  
وضع Titan: Routing حالي: اسم الأمر + نوع الحدث. لا تركيب، لا فلترة متقدمة. سؤال مفتوح: هل هذا كافٍ لنطاق Titan المستهدف، أم أن تصميم on("message") يكشف نقصاً فعلياً؟  
  
3. Conversation State & Multi-step Flows  
المشكلة الحقيقية: ConversationHandler في PTB هو الأكثر شكوى في تاريخ المكتبة: تداخل، race conditions، انتهاء صلاحية، persistence معقدة. aiogram FSM أفضل لكنه لا يزال يُربك المبتدئين. المشكلة الجذرية: إطارات أخرى بنت conversation state كطبقة فوق handler stateless، بدلاً من حلّ الـ async sequencing نفسه.  
  
وضع Titan: ask() هو جواب مختلف جذرياً — async future بدلاً من state machine. التحقق المطلوب: هل ask() يغطي الحالات التي دفعت aiogram وPTB لبناء FSM كامل؟ أم أن هناك أنماطاً حقيقية لا يغطيها؟  
  
4. API Evolution & Unknown Update Types  
المشكلة الحقيقية: Telegram أضاف في 2023-2024: message reactions، boosts، business messages، و giveaways. إطارات لم تُحدَّث بسرعة كسرت بوتات في production لأن routing لم يتعرف على الـ update_type الجديد — سقط أو رمى exception. PTB حلّها بـ default_handlers. aiogram بـ UpdateType.ANY.  
  
وضع Titan: يستحق فحصاً: ماذا يحدث عندما يصل update بنوع لا يعرفه Titan؟ هل يُهمَل بصمت، هل يرمي خطأ، هل يصل لـ error handler؟ هذا سؤال تنفيذي قصير لكن معمارياً مهم.  
  
5. Middleware Granularity  
المشكلة الحقيقية: PTB في الإصدارات الأولى لم يكن لديه middleware على الإطلاق. المطورون عوّضوا بـ decorators مخصصة ثم تشابك الكود. حين أُضيف middleware جاء متأخراً وبقيود. aiogram بنى outer/inner middleware منذ البداية لأن الاستخدام أظهر الحاجة لـ: pre-handler context enrichment، post-handler actions، per-router isolation.  
  
وضع Titan: Middleware حالي: global، linear، single next(). هذا قرار مقصود مُوثَّق. المراجعة هنا ليست "هل نضيف middleware أقوى؟" بل "هل القيود الحالية تُنتج نفس الـ workarounds التي أجبرت PTB على التغيير؟"  
  
6. Rate Limiting & Flood Control  
المشكلة الحقيقية: Telegram: 30 رسالة/ثانية globally، 1 رسالة/ثانية per chat. البوتات في production تصطدم بهذا فوراً مع أي نشاط حقيقي. PTB أضاف rate limiter ثم أزاله ثم أعاد جزءاً منه لأن الموقع الصحيح للـ throttling لم يُحسم معمارياً. aiogram يتركه لـ middleware طرف ثالث.  
  
وضع Titan: لا rate limiting. السؤال الجذري: هل هذا "خارج نطاق Core" بقرار واعٍ، أم فجوة حقيقية؟ الجواب يحدد إذا كان ADR مطلوباً أم توثيق موقف فقط.  
  
7. Webhook vs Polling — Production Deployment  
المشكلة الحقيقية: Long polling لا يعمل على معظم منصات الـ PaaS (Render, Railway, Fly.io بدون worker dyno). كل إطار ناضج بنى webhook support لأن المطورين يحتاجونه في production. PTB يوفر ASGI application. aiogram يتكامل مع aiohttp/FastAPI.  
  
وضع Titan: Polling only. هذا قد يكون حداً مقصوداً لـ v1، أو قد يكون فجوة تمنع الاعتماد في production. المراجعة تحدد: هل يوجد تصميم يُضاف لاحقاً بدون كسر العقد الحالي؟  
  
8. Application-level State & Context Enrichment  
المشكلة الحقيقية: PTB أضاف context.bot_data، context.user_data، context.chat_data لأن المطورين احتاجوا وصولاً لـ database connections وconfig من داخل الـ handlers ولم يجدوا طريقاً نظيفاً. aiogram حلّها بـ Dependency Injection في parameters. الحاجة الحقيقية: الـ handler يحتاج "شيئاً" غير موجود في الـ update.  
  
وضع Titan: ctx update-scoped فقط. كيف يصل handler لـ database session أو config object؟ حالياً عبر closure أو global — وهذا يُنتج نفس المشاكل التي دفعت PTB لإضافة bot_data.

الخلاصة:  
  
المشكلة الحقيقية — التي أثارت تقارير في مجتمعات PTB وaiogram عند إضافة message_reaction عام 2024 — موجودة في Titan بشكل مطابق. الـ _handle_update تنتهي دائماً بـ dispatch("message", ctx)، وهذا يعني أن أي update لا يعرفه BotApiTranslator (17 نوعاً من أصل 23 في Telegram API الحالية) يصل لكل handler مسجَّل على on("message") مع ctx حيث كل شيء None.
