# 015 — Data Lifecycle Responsibility

**Status:** Accepted

---

## Proposal

تحديد موقف Titan من مسؤوليتها عن البيانات التي تنشئها وتديرها داخلياً
بالنيابة عن المطوّر.

السؤال لم يُطرح صراحةً في أي قرار سابق — وغيابه أنتج التباساً في
كيفية تصنيف بيانات Links وAskManager وأي module مستقبلي يخزّن بيانات
داخل Titan.

---

## Investigation

راجع: [`docs/internal/investigations/user-privacy-erasure.md`](../internal/investigations/user-privacy-erasure.md)

التحقيق كشف أن الالتباس الأساسي لم يكن "كيف نحذف البيانات؟" بل
"ما أنواع البيانات التي تديرها Titan، وهل هي جنس واحد؟"

الإجابة: لا. Titan تدير ثلاثة أجناس مختلفة تماماً:

**أ. بيانات عبور (Transient)**
تعيش طوال معالجة update واحد فقط — `ctx`، `Update`، النماذج.
لا تتراكم، لا تُخزَّن. سؤال المسؤولية لا ينطبق عليها.

**ب. Permanent Resource Identity**
`titan_id` في Message Links Protocol. ليس "تخزيناً بدورة حياة" —
هو معرّف دائم لمورد أنشأه البوت، مكتوب صراحةً في الكود:

> *"الهوية تصف لحظة وجود — الرسالة كانت، وهذه الحقيقة لا تُمحى."*
> — `identity.py`

بالضبط كـ Git commit hash أو رابط رسالة Telegram: المحتوى قد يختفي،
الهوية تبقى. `deleted` flag يُغيّر حالة المورد — لا يمس المعرّف.
لا يوجد مفهوم "حذف رابط" داخل العقد المعماري.

**ج. User Data**
بيانات مرتبطة بمستخدم حقيقي — `AskManager._pending`، وأي module
مستقبلي يُعلن أنه يخزّن User State. هذه وحدها موضع سؤال المسؤولية.

---

## Decision

### ١. Titan تقبل المسؤولية عن دورة حياة User Data التي تنشئها داخلياً

> **Titan مسؤولة عن دورة حياة البيانات التي تُنشئها وتُديرها داخلياً
> بالنيابة عن المطوّر. هذه المسؤولية لا تمتد لبيانات المطوّر الخاصة.**

هذا القرار يجعل ما يلي إلزامياً لا اختيارياً:
- إعلان كل User Data التي تخزّنها Titan
- توفير آلية محو حقيقي — لا flags
- تعريف سلوك الاحتفاظ
- واجهة شفافية تُمكِّن المطوّر من معرفة ما تحتفظ به Titan

### ٢. التصنيف يحدد العقد — لا العكس

| النوع | المثال | المسؤولية |
|---|---|---|
| Transient | `ctx`, `Update`, النماذج | لا ينطبق — تموت بنهاية الـ update |
| Permanent Resource Identity | `message_identity`, `titan_id` | عقد بروتوكول — ليس عقد خصوصية |
| User Data | `AskManager._pending`, أي Module يُعلنه | **Titan مسؤولة — lifecycle كامل** |

### ٣. حدود المسؤولية — مكتوبة صراحةً

**تتحمله Titan:**
- كل User Data تُنشئها أو تديرها داخلياً
- إعلان كل Permanent Resource Identity وطبيعتها للمطوّر
- توفير `erase_user` كعقد عام يغطي كل User Data المُسجَّلة

**لا تتحمله Titan:**
- قواعد بيانات المطوّر الخاصة
- خدمات خارجية يرسل إليها المطوّر البيانات
- كود المطوّر الذي يستخدم `ctx.raw` أو APIs مباشرة
- `bot.banned_users` — المطوّر وضعها، المطوّر يديرها

هذا الحد الفيزيائي لا يمكن لأي framework تجاوزه تقنياً — ومحاولة
ادّعاء خلافه ستُنتج وعوداً كاذبة.

### ٤. Permanent Resource Identity خارج سياسة الخصوصية

Links ليست موضوع هذا القرار. `message_identity` لا تُصنَّف ضمن User
Data ولا تخضع لـ `erase_user` — لأنها تصف رسائل البوت لا بيانات
المستخدم. سؤال نطاق تفعيل Links هو قرار بروتوكول مستقل (ADR-018).

### ٥. الديمومة قرار تصميم — ليست فشل retention

التصنيف "Permanent Resource Identity" لا يعني "بيانات نسينا أن نحذفها"
أو "احتفاظ غير مبرر". الديمومة هنا تصف حقيقةً تاريخية لا يمكن محوها:

> *البوت أرسل هذه الرسالة. هذا الفعل كان.*

`titan_id` دائم بنفس المنطق الذي يجعل Git commit hash دائماً — المحتوى
قد يُحذف، العنوان لا. علامة `deleted` تُغيّر حالة المورد ("لم تعد
الرسالة متاحة") ولا تمس الهوية ("الرسالة 482 كانت").

الفرق بين النوعين مقصود ومكتوب:

| | Permanent Resource Identity | User Data |
|---|---|---|
| ماذا تصف؟ | فعلاً تاريخياً لا يُنسخ | حالة مستخدم تتغير |
| الديمومة | قرار تصميم — تعبير عن حقيقة | غير مبررة بدون عقد احتفاظ |
| الحذف | تغيير في حالة المورد، لا في هويته | محو حقيقي — شرط أساسي |

"Permanent" في هذا السياق ليس نقيض "قابل للحذف" — هو نقيض "مؤقت بطبعه".
Transient تموت تصميماً، User Data تُحذف بطلب، Permanent Resource
Identity تُحتفظ بها لأن وجودها يمثّل حقيقة لا يُستحسن إنكارها.

---

## Rule

**كل بيانات مستمرة تُضاف إلى Titan تُصنَّف في وقت التصميم.**

التصنيف يحدد العقد الذي ينطبق عليها:
- Permanent Resource Identity → عقد بروتوكول (ما معنى وجوده، ما معنى `deleted`)
- User Data → عقد دورة حياة (إعلان، محو، شفافية)

Module لا يُصنِّف بياناته يحمل ديناً معمارياً صريحاً.

---

## Alternatives Considered

**Titan لا تتحمل مسؤولية — توفر أدوات فقط**

يعني: `erase_user` اختيارية، `/mydata` اختيارية، المطوّر يقرر.

لم يُختر لأنه متناقض: Titan تخزّن تلقائياً بالنيابة عن المطوّر —
ثم تقول "أنت مسؤول عن الحذف." من قرّر التخزين يملك المسؤولية.

**Titan مسؤولة عن كل بيانات البوت بما فيها قواعد بيانات المطوّر**

يعني: Titan تضمن محو كل شيء عند `/forgetme`.

لم يُختر لأنه مستحيل تقنياً وسيُنتج وعوداً كاذبة. Framework لا
يملك رؤية كاملة لكل ما يكتبه المطوّر خارجه.

---

## Consequences

**ما يُكتسب:**
- `erase_user` أصبح ضرورة مشتقة من مسؤولية مُعلنة — لا ميزة إضافية.
- `data_held_for` أصبح متوقعاً لا ترفاً.
- كل module جديد في Titan يعرف مسبقاً ما يُلزمه به.
- المستخدم النهائي يملك ضماناً: ما تخزّنه Titan عنه قابل للمحو.

**القيود المقبولة:**
- ما يخزّنه المطوّر خارج Titan خارج الضمان — موثَّق صراحةً.
- Titan لا تمنع مطوّراً يتجاهل `erase_user` عمداً — تجعل تجاهله
  قراراً واعياً لا جهلاً.

**يُشرِّع مباشرةً:**
- ADR-016: User Data Registry و`erase_user` architecture
- ADR-017: الأوامر المحجوزة `/mydata` و`/forgetme`
